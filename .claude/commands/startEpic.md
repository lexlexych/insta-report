# Эпик-режим

Этот файл — регламент эпик-режима. Он ДОПОЛНЯЕТ docs/orchestration/workflow.md и с санкции
человека отступает от него ровно в двух местах: (1) нет пер-тикетного СТОПа — тикеты эпика идут
подряд в ОДНОЙ ветке `epic/E<N>`; (2) один PR на весь эпик в конце, поэтому у тикета в
awaiting_merge поле pr временно = null. ВСЁ остальное из workflow.md действует: запреты, статусы,
обязательный docs/manual-tests/T-0XX.md до финализации, максимум 2 цикла ревью, done ставит только человек.

Роли: ты (Opus) — оркестратор, код НЕ пишешь. Реализация — Codex CLI (`codex exec`), НЕ сабагент
implementer. Ревью — сабагент reviewer.

## 0. Preflight
1. Полный RESUME из workflow.md (fetch, checkout main, pull; `chore: sync progress` при изменениях).
2. `codex --version`. CLI нет или не отвечает → СТОП: предложи человеку установить codex либо явно
   разрешить fallback на сабагент implementer для всего эпика. Сам не решай.
3. Эпик: из $ARGUMENTS; пусто → эпик следующего доступного pending-тикета (deps выполнены, меньший номер).
4. Состав эпика — ТОЛЬКО по колонке «Эпик» в docs/tickets/00-INDEX.md (значение «N» или «N <название>»).
   deps каждого тикета бери из state/progress.json (при расхождении с INDEX истина — progress.json).
5. Раздели тикеты эпика: done/skipped — пропустить; pending — в работу; awaiting_merge на ветке
   НЕ этого эпика, failed, blocked, in_progress вне эпик-ветки → смешанное состояние, СТОП и вопрос
   человеку (варианты: сначала влить старые PR / переоткрыть тикет в эпике / пропустить).
   Все тикеты done|skipped → доложи «эпик завершён» и предложи следующий.
6. Порядок: топологическая сортировка по deps внутри эпика, при равенстве — меньший номер.
7. Внешние deps (вне эпика): каждая должна быть done или skipped (skipped считается
   удовлетворённой по OVERRIDES — упомяни это в отчёте). Иначе → СТОП, доложи, статусы не менять.
8. Если в $ARGUMENTS есть «plan» → выведи план (порядок тикетов, deps, состояние ветки/прогресса)
   и ОСТАНОВИСЬ, ничего не выполняя и не меняя.

## 1. Эпик-ветка
- Имя строго `epic/E<N>` (например epic/E10). Push только явной формой `git push origin epic/E<N>`.
- Ветки нет → от свежего main: `git checkout -b epic/E<N>`.
- Ветка есть (прерванный прогон) → checkout; если есть origin/epic/E<N> — pull. Грязное рабочее
  дерево → СТОП, спроси человека (чьи это правки — неизвестно). Возобновление по progress.json:
  - тикеты с notes «ЭПИК E<N>…» и статусом awaiting_merge — завершены в эпике, пропусти;
  - тикет in_progress: base = `git log --format=%H --grep="^T-0XX: старт" -n 1`; если после base
    есть коммиты `T-0XX:` — продолжай с шага 4 (самопроверка → ревью), иначе — с шага 3 (реализация).
- Эпик после обрыва возобновляй ТОЛЬКО через /startEpic <N>, не через /ticket.

## 2. Старт тикета (для каждого тикета по порядку)
1. deps-check: done | skipped | «завершён в этой эпик-ветке» (awaiting_merge + branch epic/E<N>).
2. progress.json: status in_progress, branch "epic/E<N>", attempts +1,
   notes "ЭПИК E<N>: старт <дата>". Коммит ТОЛЬКО этого файла: `T-0XX: старт в эпике E<N> (in_progress)`.
3. Запомни базу ревью: `BASE=$(git rev-parse HEAD)`.

## 3. Реализация — Codex
Запуск (длинный промпт — через stdin; команда долгая — запускай в фоне и жди, бюджет 45 минут):
```
codex exec --full-auto --output-last-message /tmp/codex-T-0XX.md - <<'PROMPT'
Ты — implementer тикета T-0XX проекта InstaReply (репозиторий — текущая директория).
1. Прочитай ЦЕЛИКОМ: docs/tickets/T-0XX.md, docs/tickets/OVERRIDES.md, docs/plan.md §5,
   docs/orchestration/manual-testing.md. OVERRIDES сильнее тикета. Тестовые файлы НЕ создавать;
   сценарии раздела «Тесты» тикета обязан учесть как краевые случаи в коде.
2. Изучи точечно только модули, на которые тикет опирается.
3. Реализуй по шагам тикета. Меняй только файлы, вытекающие из тикета (+ scripts/manual/* при
   необходимости demo-скрипта). Заглушки — только "// TODO(T-0YY)".
4. Создай docs/manual-tests/T-0XX.md строго по шаблону docs/orchestration/manual-testing.md.
5. Прогони `pnpm lint` и `pnpm typecheck`, исправь до зелёного.
ЗАПРЕЩЕНО: любые git-команды; правки state/**, .claude/**, docs/** (кроме docs/manual-tests/T-0XX.md);
чтение .env*; новые зависимости, не названные в тикете.
Финальное сообщение СТРОГО: STATUS completed|failed; FILES (список); EDGE_CASES (сценарий →
файл:функция); QUESTIONS.
PROMPT
```
- Отчёт читай из /tmp/codex-T-0XX.md, НЕ из полного stdout.
- Крэш/таймаут/пустой отчёт → один повтор. Вторая неудача → процедура FAILED (раздел 8).
- Если `--full-auto` не поддерживается установленной версией → `--sandbox workspace-write`.
  Сетевой доступ codex НЕ включать.

## 4. Самопроверка оркестратора (не верь отчёту)
1. `pnpm lint && pnpm typecheck` — сам. Красное → до 2 fix-вызовов Codex (промпт: «режим fix,
   тикет T-0XX, исправь ошибки lint/typecheck: <вывод>», те же запреты и формат отчёта). Затем FAILED.
2. docs/manual-tests/T-0XX.md существует и не пуст. Нет → fix-вызов Codex.
3. `git status`: несанкционированные правки (state/**, docs/** кроме manual-tests, .claude/**,
   package.json без указания тикета) → откати `git checkout -- <файл>` и отметь в notes.
4. Коммит: `T-0XX: <название тикета>` (код + manual-test).

## 5. Ревью — сабагент reviewer
Промпт диспатча: «Тикет T-0XX. ВАЖНО: ветка эпика содержит несколько тикетов, поэтому вместо
`git diff main...HEAD` ревьюй ТОЛЬКО диапазон этого тикета: `git diff <BASE>..HEAD`
(BASE=<хеш>). Остальное — по твоей инструкции.»
- changes_required → Codex в режиме fix с дословным списком BLOCKERS → коммит
  `T-0XX: правки по ревью (цикл N)` → lint/typecheck → повторное ревью того же диапазона.
- Максимум 2 цикла доработки, дальше — FAILED (раздел 8).

## 6. Финализация тикета
1. progress.json: status awaiting_merge, branch "epic/E<N>", pr null, notes
   "ЭПИК E<N>: завершён в эпик-ветке <дата>. Review: approved (<K> циклов). lint+typecheck зелёные.
   manual-test: docs/manual-tests/T-0XX.md. PR эпика будет открыт по завершении эпика."
2. docs/build-log.md: строка `| T-0XX | awaiting_merge | epic/E<N> | PR эпика (в конце) | K | дата |`.
3. Коммит `T-0XX: финализация (awaiting_merge)`. Затем `git push origin epic/E<N>`
   (2 повтора при сбое; не получилось — продолжай локально, отметь для финального отчёта;
   но push перед PR в разделе 7 обязателен).
4. Переходи к следующему тикету (раздел 2). Пер-тикетного СТОПа в эпик-режиме НЕТ.

## 7. Эпилог (все тикеты эпика awaiting_merge)
1. Финальный `git push origin epic/E<N>` — обязателен, при стойком сбое СТОП и доложи.
2. Собери тело PR в /tmp/pr-epic-E<N>.md: сводная таблица (тикет | название | циклы ревью);
   затем для каждого тикета: 2-3 строки что сделано, чек-лист критериев приёмки,
   ПОЛНЫЙ текст docs/manual-tests/T-0XX.md.
3. `gh pr create --base main --head epic/E<N> --title "Epic E<N>: <название> (T-0AA…T-0ZZ)" --body-file /tmp/pr-epic-E<N>.md`.
4. progress.json: pr = URL для ВСЕХ тикетов эпика; build-log: итоговая строка эпика с PR.
   Коммит `chore(E<N>): ссылки на PR в progress.json` (допустимое отступление от префикса T-0XX), push.
5. СТОП. Отчёт человеку: ссылка на PR, таблица тикетов, рекомендуемый порядок ручного
   тестирования (3-5 строк). Следующий эпик не начинать.

## 8. FAILED — остановка эпика
Ревью не approved после 2 циклов, Codex дважды не справился, lint не чинится или иная
неустранимая проблема:
1. Незакоммиченные правки закоммить как `T-0XX: WIP (эпик остановлен)` — ничего не терять.
2. progress.json: status failed, notes — ПОЛНЫЙ диагноз (что делалось, вердикты и BLOCKERS ревью,
   вывод lint/typecheck, гипотезы). Коммит `T-0XX: failed (диагноз в progress.json)`.
3. `git push origin epic/E<N>` — запушить всё сделанное.
4. PR НЕ открывать. Остальные тикеты эпика не трогать (остаются pending).
5. СТОП. Отчёт: какие тикеты завершены в эпик-ветке, что провалилось и почему, варианты
   (уточнить тикет / доделать вручную / пропустить / перезапустить эпик после правки).

не force-push, не reset --hard, не rm -rf; done не ставить — это делает человек; новые зависимости
только по тикету.
