# Настройка: разработка в облаке через Claude Code ИЛИ Codex (взаимозаменяемо)

## Идея
Оба агента работают с одним GitHub-репозиторием и одним набором правил. Claude Code читает
`CLAUDE.md`, Codex читает `AGENTS.md` — оба файла тонкие и ссылаются на общий регламент
`docs/orchestration/workflow.md`. Всё состояние — в репозитории (`state/progress.json`),
поэтому сессию, начатую в Claude, можно продолжить в Codex и наоборот: закончились токены
Opus → открываешь Codex → «выполни RESUME и продолжай».

Автотестов нет. Гейт качества — ты: каждый тикет = ветка `ticket/T-0XX` + PR, в PR — инструкция
ручного теста (`docs/manual-tests/T-0XX.md`). Протестировал локально → влил → агент берёт следующий.

## Структура

```
├── CLAUDE.md                      # правила для Claude Code (Opus-оркестратор + Sonnet-сабагенты)
├── AGENTS.md                      # правила для Codex (роли implementer→reviewer→scribe последовательно)
├── SETUP.md
├── .claude/
│   ├── settings.json              # model: opus; разрешения: push только в ticket/*, deny .env
│   ├── commands/                  # /ticket, /status, /resume (обёртки над workflow.md)
│   └── agents/                    # implementer (Sonnet), reviewer (Sonnet)
├── docs/
│   ├── plan.md                    # план проекта + блок «Ревизия v1.1» (без автотестов, без MCP)
│   ├── orchestration/
│   │   ├── workflow.md            # ЕДИНЫЙ регламент: статусы, конвейер, RESUME, git/PR, запреты
│   │   └── manual-testing.md      # шаблон инструкции ручного теста
│   ├── tickets/                   # 00-INDEX + T-001..T-032 + OVERRIDES.md (сильнее тикетов)
│   ├── manual-tests/              # создаются агентами по мере выполнения
│   └── build-log.md
└── state/progress.json            # статусы (в т.ч. awaiting_merge), ветки, PR, deps
```

## Запуск через Claude Code (облако или CLI)
1. Создай GitHub-репо, залей содержимое, дай Claude Code доступ к репо.
2. В сессии: `/status` → увидишь 31 pending + T-031 skipped.
3. `/ticket` (или `/ticket T-001`) → агент сделает ветку, код, ревью, manual-test, PR и остановится.
4. Ты: `git fetch && git checkout ticket/T-001`, пройди `docs/manual-tests/T-001.md`, влей PR.
5. `/resume` → статус done → `/ticket` следующий. Повторяй.

## Запуск через Codex (когда кончились токены Claude — или просто вместо него)
1. Codex автоматически читает `AGENTS.md` в корне репо (cloud-задача или `codex` в клоне репо).
2. Скажи: «Выполни процедуру RESUME из docs/orchestration/workflow.md и возьми следующий тикет».
3. Дальше цикл тот же: PR → твой ручной тест → merge → следующий.
Удобно завести себе сохранённый промпт (например, `~/.codex/prompts/next-ticket.md`) с этой фразой.

## Переключение Claude ↔ Codex
Ничего специального: перед уходом агент обязан (workflow.md, «Передача смены») докоммитить шаг,
запушить ветку и обновить progress.json. Принимающий начинает с RESUME. Если сессия оборвалась
жёстко (лимит посреди работы) — RESUME обнаружит in_progress и спросит тебя: продолжить ветку
или начать тикет заново.

## Локальная среда для ручных тестов (одноразовая настройка)
- Node 20+, pnpm; `.env` по `.env.example` (появится после T-001).
- Dev-проект Supabase (регион EU) — миграции применяешь ты: `pnpm db:push`.
- Тестовый Telegram-бот (BotFather) и туннель для вебхуков (`npx localtunnel` / cloudflared) —
  понадобятся с T-005/T-011; каждая инструкция manual-test напоминает, что именно нужно.

## Правила, которые стоит знать
- done ставится только после мержа — агенты не имеют права «принять» работу сами.
- Прямые коммиты в main запрещены (кроме служебного sync progress).
- Тексты тикетов и регламент — read-only для агентов; править их можешь только ты
  (точка расширения — docs/tickets/OVERRIDES.md).
- Сценарии из разделов «Тесты» в тикетах не выбрасываются: implementer обязан учесть их как
  краевые случаи кода, а ключевые попадают в шаги твоей ручной проверки.
