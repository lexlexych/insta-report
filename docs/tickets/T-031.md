# T-031 · E2E смоук-тест сценариев

> **Перед началом:** прочитай `docs/plan.md` §2.2, §5. Зависимости: T-022 (весь пайплайн). Все внешние API мокируются на границе: fetch (IG, Telegram, LLM) и db-клиент.

## Цель
Интеграционные тесты «полного круга», прогоняющие реальные модули пайплайна вместе.

## Шаги выполнения
1. Инфраструктура `tests/e2e/harness.ts`:
   - in-memory реализация db-слоя: замокай `src/lib/db/*` фейками на Map (tenants, connections c зашифрованными секретами через реальный crypto, drafts c честной семантикой claim/partial-unique, processed_events);
   - мок fetch c маршрутизацией по URL: graph.instagram.com/* → программируемые ответы; api.telegram.org/* → запись вызовов (sendMessage возвращает инкрементный message_id); LLM_BASE_URL → фиксированные ответы classify/draft;
   - фабрика `makeTenant()` — заполненный тенант c активным подключением;
   - хелперы: `postIgWebhook(tenantId, fixture, appSecret)` — собирает Request c ВАЛИДНОЙ подписью по raw-телу и вызывает реальный POST-роут T-011; `pressButton(callbackData)` — скармливает боту update callback_query.
2. Сценарий A «happy path»: postIgWebhook(incoming_text) → assert: TG sendMessage вызван, карточка содержит черновик из мока LLM, в drafts один pending → pressButton('send:{id}') (getConversation мок: без ручного ответа) → assert: IG /messages вызван c draft_text, статус sent, editMessage карточки на месте c «✅ отправлено в …», message_log имеет in и out.
3. Сценарий B «echo отменяет»: incoming → pending → postIgWebhook(echo от аккаунта бизнеса) → assert: deleteMessage вызван, статус cancelled, IG /messages НЕ вызывался.
4. Сценарий C «дубликат вебхука»: два postIgWebhook c одним mid → ровно один sendMessage, один pending.
5. Сценарий D «ручной ответ между карточкой и кнопкой»: pending; getConversation теперь отдаёт сообщение бизнеса c createdTime > trigger_ts → pressButton → статус skipped_manual, IG send не вызван.
6. Сценарий E «второе сообщение клиента замещает черновик»: incoming №1 → pending₁; incoming №2 (другой mid) → deleteMessage(карточка₁), pending₂ единственный.
7. Подключи e2e в CI отдельным шагом `pnpm test:e2e` (vitest --dir tests/e2e).

## Критерии приёмки
- [ ] Все 5 сценариев зелёные и запускаются в CI
- [ ] Подпись вебхука в harness настоящая (реальный код проверки задействован)
- [ ] Фейковый db честно эмулирует атомарность claim (двойной pressButton в сценарии A → одна отправка)
