# OVERRIDES — обязательные поправки к тикетам (ревизия v1.1)

Этот файл сильнее текста тикетов. Конфликт → действует OVERRIDES. Менять может только человек.

## 1. Автоматические тесты исключены из проекта
- Разделы «Тесты» во ВСЕХ тикетах НЕ выполнять: тестовые файлы не создавать, vitest не ставить.
- Каждый пункт «Критериев приёмки», ссылающийся на тест, читать как «поведение демонстрируется
  ручной проверкой из docs/manual-tests/T-0XX.md».
- ВНИМАНИЕ: перечисленные в разделах «Тесты» сценарии — это ценные списки краевых случаев.
  Implementer обязан УЧЕСТЬ их в коде, а scribe — превратить ключевые из них в шаги ручной проверки.
- Definition of Done везде: `pnpm lint && pnpm typecheck` (без pnpm test).

## 2. Правки конкретных тикетов
- T-001: Vitest и tests/ не создавать; CI-workflow — только lint + typecheck; каталог
  scripts/manual/ добавить в структуру; README-раздел «Ручное тестирование» со ссылкой на
  docs/manual-tests/.
- T-002: вместо теста на содержимое миграций — ручная проверка: применить на чистый dev-проект
  Supabase и приложить в manual-test чек-лист «какие таблицы/индексы/функции должны появиться».
- T-004, T-008, T-013, T-016–T-022: сценарии из разделов «Тесты» реализовать как краевые случаи
  кода; для модулей без UI — demo-скрипты в scripts/manual/ (см. шаблон manual-testing.md).
- T-025: лимит симулятора проверяется вручную (снизь до 3/день переменной окружения
  SIMULATOR_DAILY_LIMIT — добавь её в env.ts с дефолтом 30).
- T-029: правило no-console оставить (это lint, не тест).
- T-030: «аудит-тест роутов» заменить на скрипт `scripts/manual/audit-routes.ts`
  (тот же анализ, запуск руками; добавь его шагом в manual-test тикета).
- T-031: ОТМЕНЁН (status: skipped) — e2e-автотесты исключены. Его 5 сценариев перенесены
  в docs/manual-tests/T-022.md как расширенный ручной прогон (implementer T-022 обязан их включить).
- T-032: зависимость меняется с T-031 на T-030; в docs/deploy.md чек-лист первого тенанта
  дополнить сводной ссылкой на все manual-tests.

## 3. Инфраструктура
- Supabase MCP не используется. Вся работа с БД — файлы миграций + `pnpm db:push` человеком
  на его dev-проекте (агенты команды к живой БД не выполняют).
- Разработка в облаке: агенты пушат ветки ticket/T-0XX и открывают PR; человек тестирует локально
  и мержит. Прямые коммиты в main запрещены (см. workflow.md).
