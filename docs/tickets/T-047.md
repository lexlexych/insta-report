# T-047 · Zernio: бэкенд подключения (connect / callback / status / disconnect)

> **Перед началом:** прочитай `docs/research/zernio.md` §4–5 и `docs/plan.md` §5. Посмотри
> реализацию Meta-аналогов: `app/api/miniapp/ig/login-url/route.ts`, `app/api/ig/callback/route.ts`,
> `src/lib/ig/oauthState.ts`. Зависимости: T-045, T-046.
> Эпик 12 «Instagram через Zernio», ветка эпика — `zernio`.

## Цель
Полный серверный цикл подключения Instagram через Zernio: выдача OAuth-URL из Mini App,
приём redirect'а, привязка аккаунта к тенанту, статус и отключение. Без шагов «заявка →
подтверждение админом» и «приглашение тестировщика» — в этом весь смысл эпика.

## Шаги выполнения
1. **Подписанный state** `src/lib/zernio/state.ts`: переиспользуй механику
   `src/lib/ig/oauthState.ts` (HMAC-SHA256 на ключе из `getKey()`, TTL 15 минут) —
   либо обобщи существующий модуль (не ломая `ig/callback`), либо создай параллельный
   с payload `{ tenantId, embedded, nonce, iat }`. В БД ничего не хранить.
2. **`POST /api/miniapp/zernio/connect`** (requireTenant, `apiHandler`):
   - `!isZernioEnabled()` → 404 `{ ok: false, error: 'disabled' }`;
   - `zernioAccounts.getForTenant`: нет записи → `createProfile('tenant-' + tenant.id)`
     (имя ≤ 50 симв.) и `insertPending`; есть → взять её `zernio_profile_id`;
     статус `active` → 409 `{ error: 'already_connected' }` (сначала отключить);
   - `redirect_url` = `${env.APP_BASE_URL}/api/zernio/callback?state=<sign({tenantId, embedded})>`
     — Zernio дописывает свои параметры к существующей query (research §4), state доедет;
   - `getConnectUrl('instagram', profileId, redirectUrl)` → ответ `{ ok: true, url: authUrl }`;
   - тело запроса: `{ embedded?: boolean }` (как в `login-url?embedded=1` — для мобильного TG);
   - ошибки Zernio (403 лимит профилей, 401) → 502 `{ error: 'zernio_error' }`,
     подробности в серверный лог БЕЗ api-ключа.
3. **`GET /api/zernio/callback`** — публичный роут (авторизация = валидный state, по образцу
   `app/api/ig/callback/route.ts`, включая HTML-страницы и `miniappRedirect`):
   - невалидный/просроченный `state` → 403, никаких вызовов Zernio;
   - параметры успеха от Zernio: `connected=instagram&profileId=X&accountId=Y&username=Z`;
     нет `accountId` → страница «Не получилось» (см. ниже; research §9.1 — форма отказа
     не документирована, отсутствие accountId трактуем как отказ/ошибку);
   - защита от подделки query: `profileId` из query обязан совпадать с
     `zernio_accounts.zernio_profile_id` тенанта из state; затем **серверная сверка** —
     `listAccounts({ profileId })` через клиент: `accountId` присутствует и
     `platform === 'instagram'`. Иначе 403 без записи в БД;
   - happy path: `zernioAccounts.activate(tenantId, 'instagram', { zernioAccountId,
     username: username ?? найденный в listAccounts })`;
   - идемпотентность: повторный заход по тому же redirect'у → снова activate, дубликатов нет;
   - `embedded` из state → redirect в мини-апп (`miniappRedirect('/app?zernio=success')`),
     иначе — HTML-страница «Instagram @username подключён через Zernio» со ссылкой
     `https://t.me/{TELEGRAM_BOT_USERNAME}?startapp=zernio` (username в HTML — только через
     `escapeHTML`); ошибка → страница «Не получилось, попробуйте ещё раз» + `?zernio=error`
     для embedded;
   - ошибки Zernio API на сверке → страница ошибки, `setStatus(..., 'error', <причина>)`;
     в логи — без ключа и без полного authUrl.
4. **Deep-link возврата**: в `src/components/miniapp/TmaProvider.tsx` добавь обработку
   `tgWebAppStartParam === 'zernio'` → `router.replace('/app')` (дашборд; по образцу
   существующего `'connect'`).
5. **`GET /api/miniapp/zernio/status`** (requireTenant):
   - `{ ok: true, enabled: isZernioEnabled(), status: 'none' | 'pending' | 'active'
     | 'disconnected' | 'error', username, connectedAt }` — из `zernio_accounts`
     (нет записи → `none`); сетевых вызовов НЕ делать (быстрый poll с дашборда).
6. **`POST /api/miniapp/zernio/disconnect`** (requireTenant): запись есть и был
   `zernio_account_id` → `deleteAccount(zernioAccountId)` (404 от Zernio игнорировать —
   уже удалён), затем `zernioAccounts.disconnect(tenantId, 'instagram', 'user_request')` →
   `{ ok: true }`. По образцу `app/api/miniapp/ig/disconnect`.
7. **Краевые случаи**: два тенанта подключают один и тот же IG-аккаунт через Zernio
   (unique по `zernio_account_id` → второй получает страницу ошибки «аккаунт уже подключён
   в другом кабинете»); повторный `connect` после `disconnected` (профиль переиспользуется,
   новый не создаётся); Zernio недоступен (таймаут) на каждом из роутов.
8. **Manual-test** `docs/manual-tests/T-047.md`: подготовка — env из T-045, `pnpm db:push`
   (T-046), туннель к dev-серверу (для callback нужен публичный APP_BASE_URL); проверки —
   connect возвращает валидный authUrl; полный OAuth-проход в браузере → callback →
   страница успеха → строка в `zernio_accounts` со статусом active; повторный проход
   идемпотентен; disconnect удаляет аккаунт в дашборде Zernio; негативные — битый state →
   403, подмена profileId в query → 403.

## Вне объёма
- Вебхуки `account.*` (T-048), пайплайн сообщений (T-049/T-050), UI-панель (T-051).
- Экран-визард с промежуточными шагами — НЕ нужен (одна кнопка на дашборде, T-051).

## Критерии приёмки
- [ ] Подключение проходит без подтверждения админом и без приглашения тестировщика:
      кнопка → OAuth-экран → страница успеха → `zernio_accounts.status='active'`
- [ ] Callback не доверяет query: подпись state + сверка profileId + серверная проверка
      аккаунта через `listAccounts`
- [ ] Callback идемпотентен; чужой/просроченный state → 403 без вызовов Zernio и записи в БД
- [ ] API-ключ Zernio и state не попадают в логи, ответы и HTML
- [ ] При `!isZernioEnabled()` все новые роуты отвечают 404, поведение остального
      приложения не меняется
- [ ] Прямое Meta-подключение (`/api/miniapp/ig/*`, `/api/ig/callback`) не затронуто
- [ ] lint, typecheck зелёные; manual-test написан
