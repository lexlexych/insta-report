# T-002 · Схема БД и миграции Supabase

> **Перед началом:** прочитай `docs/plan.md` §2.4 (модель данных) и §5 (соглашения). Зависимости: T-001.

## Цель
Полная схема Postgres в виде миграций Supabase, RLS deny-all для anon, генерация TS-типов.

## Шаги выполнения
1. Добавь `supabase` CLI как devDependency; выполни `supabase init` (каталог `/supabase` уже существует).
2. Создай миграцию `supabase/migrations/0001_init.sql` строго по docs/plan.md §2.4. Требования сверх описанных полей:
   - Все `id uuid primary key default gen_random_uuid()`; все `created_at timestamptz default now()`.
   - `tenants.telegram_user_id bigint not null unique`.
   - `ig_connections.tenant_id` — `unique references tenants(id) on delete cascade`. Все дочерние таблицы — `on delete cascade` от tenants.
   - Статусы — тип `text` c `check`-констрейнтами по спискам значений из плана (не enum, чтобы упростить миграции).
   - Индексы: `create unique index processed_events_uniq on processed_events(tenant_id, event_mid);`
     `create unique index drafts_pending_uniq on drafts(tenant_id, conversation_key) where status = 'pending';`
     `create unique index usage_stats_uniq on usage_stats(tenant_id, day);`
     обычные индексы: `drafts(tenant_id, status)`, `message_log(tenant_id, created_at desc)`.
3. В той же миграции: `alter table ... enable row level security;` для всех таблиц. Политик не создавать (service role обходит RLS; anon получает deny-all по умолчанию).
4. Функция сида меток:
   ```sql
   create or replace function seed_default_labels(p_tenant uuid) returns void language sql as $$
     insert into labels (tenant_id, name, description, instruction, sort)
     values (p_tenant, 'Без категории', 'Обращения, не подошедшие под другие категории',
             'Ответь вежливо и по существу, опираясь на базу знаний.', 999)
     on conflict do nothing;
   $$;
   ```
   Для `on conflict do nothing` добавь `unique (tenant_id, name)` на labels.
5. Миграция `0002_increment_usage.sql` — атомарный инкремент статистики:
   ```sql
   create or replace function increment_usage(p_tenant uuid, p_day date,
     p_llm_calls int, p_tokens_in int, p_tokens_out int,
     p_drafts_created int, p_drafts_sent int) returns void language sql as $$
     insert into usage_stats as u (tenant_id, day, llm_calls, tokens_in, tokens_out, drafts_created, drafts_sent)
     values (p_tenant, p_day, p_llm_calls, p_tokens_in, p_tokens_out, p_drafts_created, p_drafts_sent)
     on conflict (tenant_id, day) do update set
       llm_calls = u.llm_calls + excluded.llm_calls,
       tokens_in = u.tokens_in + excluded.tokens_in,
       tokens_out = u.tokens_out + excluded.tokens_out,
       drafts_created = u.drafts_created + excluded.drafts_created,
       drafts_sent = u.drafts_sent + excluded.drafts_sent;
   $$;
   ```
6. Скрипты package.json: `"db:push": "supabase db push"`, `"db:types": "supabase gen types typescript --linked > src/lib/db/types.gen.ts"`. Сгенерируй `types.gen.ts` (если нет доступа к живому проекту — напиши типы вручную в том же формате `Database` и оставь TODO-комментарий о регенерации).
7. Обнови README: раздел «База данных» (как применить миграции, как регенерировать типы).
8. Тест `tests/migrations.test.ts`: прочитай SQL-файлы как текст и проверь наличие ключевых конструкций (partial unique index на drafts, RLS enable для каждой таблицы, обе функции). Это дешёвая защита от случайного удаления.
