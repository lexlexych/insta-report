# T-022 · Обработчик echo (ручной ответ владельца)

> **Перед началом:** прочитай `docs/plan.md` §2.2, §5. Зависимости: T-016, T-020.

## Цель
Ручной ответ бизнеса в Instagram отзывает висящий черновик.

## Шаги выполнения
1. `src/lib/pipeline/handleEcho.ts` (замени заглушку):
   ```
   key = conversationKey(ev.accountId, ev.contactId)
   draft = drafts.cancelPendingByConversation(tenantId, key)
   if (!draft) return                       // нечего отзывать
   await deleteMessageSafe(draft.tg_chat_id, draft.tg_message_id)
   await messageLog.add(tenantId, key, 'manual', ev.text || '[вложение]')
   ```
2. **Различение своего echo:** когда T-021 отправляет ответ через API, Meta пришлёт echo этой же отправки. Он не должен отменить СЛЕДУЮЩИЙ свежесозданный черновик. Механика: в T-021 после успешной отправки IG API возвращает mid'ы частей — сразу `processedEvents.tryInsert(tenantId, mid)` для каждого. Тогда echo своего сообщения отсечётся dedup'ом в handleIgEvent ещё до handleEcho. Внеси это дополнение в `src/lib/pipeline/send.ts` (+ убедись, что sendMessage в T-013 возвращает mid[]; если нет — доработай).
3. Тесты `tests/pipeline.echo.test.ts`:
   - есть pending → cancel + deleteMessageSafe + messageLog(manual);
   - нет pending → no-op, ни одного вызова tg;
   - deleteMessage бросил «not found» → поток не падает, статус всё равно cancelled;
   - интеграционно (моки): send из T-021 регистрирует mid'ы → последующий handleIgEvent c echo этого mid не доходит до handleEcho (tryInsert=false).

## Критерии приёмки
- [ ] Ручной ответ гарантированно убирает карточку из Telegram
