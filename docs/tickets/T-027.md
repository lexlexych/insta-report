# T-027 · Дашборд

> **Перед началом:** прочитай `docs/plan.md` §1, §5. Зависимости: T-014, T-021.

## Цель
Главная страница Mini App: метрики, статус подключения, последняя активность.

## Шаги выполнения
1. `GET /api/miniapp/dashboard` (requireTenant): параметр `days=7|30`;
   - агрегаты из usage_stats за период: llm_calls, tokens (in+out), drafts_created, drafts_sent;
   - из drafts: counts по статусам за период (cancelled+skipped_manual = «отвечено вручную»);
   - подключение: краткий объект из логики T-014 (вынеси в `src/lib/ig/status.ts` общую функцию, чтобы не дублировать);
   - `recent: messageLog.recent(tenantId, 10)` (direction, обрезанный text 80 симв., created_at).
2. UI `/app` (главная):
   - если onboarding не done → редирект (уже в T-009);
   - статус-карточка подключения: зелёная «@username · работает» / жёлтая «Требует настройки» / красная «Ошибка» — тап ведёт на connect-instagram;
   - переключатель 7/30 дней; сетка 2×2 метрик: Диалогов · Черновиков · Отправлено · Вручную; под ней строка «LLM: {calls} вызовов · {tokens} токенов»;
   - блок «Последняя активность»: список из recent c иконками направления (📩 in / ✅ out / ✋ manual) и относительным временем;
   - пустое состояние (нет активности): иллюстративный блок c CTA «Подключить Instagram» или «Открыть тест-чат» (в зависимости от статуса подключения).
3. Лёгкий поллинг: обновление раз в 30 с при видимой вкладке (visibilitychange).
4. Тесты: API-агрегация на фикстурных данных (моки repo) — цифры сходятся; drafts sent+manual+pending согласованы; UI — пустое состояние vs заполненное (мок fetch).

## Критерии приёмки
- [ ] Цифры дашборда сходятся c тестовыми данными (юнит на агрегацию)
- [ ] Каждая карточка ведёт на релевантный экран
- [ ] Пустое состояние осмысленно для нового пользователя
