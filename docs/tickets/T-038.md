# T-038 · Онбординг 2.0 — полноэкранный визард (UI + финализация)

> **Перед началом:** прочитай `docs/plan.md` §1, §5 (ревизия v1.2) и `docs/tickets/OVERRIDES.md`.
> Зависимости: T-033 (OAuth-подключение), T-035 (сферы/шаблоны), T-037 (заявки).
> Этот тикет ЗАМЕНЯЕТ визард T-009: LLM-генерации базы знаний больше нет, база знаний берётся
> из шаблона выбранной сферы деятельности и правится пользователем. Демонтаж старого кода —
> отдельным тикетом T-039 (здесь — только замена страницы онбординга).

## Цель

Полноэкранный онбординг из 5 экранов: сфера деятельности → данные бизнеса и Instagram-аккаунт →
ветка «заявка подана» / «подключение Instagram» → правка шаблона базы знаний → финальный экран
«как пользоваться системой».

## Каркас

1. Страница `app/(miniapp)/app/onboarding/page.tsx` переписывается. Пока
   `tenants.onboarding_step !== 'done'`:
   - онбординг занимает весь экран, нижняя навигация разделов СКРЫТА (гард-редиректы из T-009
     сохранить, набор значений шага обновить);
   - вместо навигации внизу — фиксированная панель с одной кнопкой «Далее» (обычная in-page
     кнопка, НЕ Telegram MainButton; прежнее использование MainButton убрать со страницы);
   - «Далее» неактивна (disabled + визуально приглушена), пока данные текущего экрана
     не заполнены/невалидны.
2. Значения `onboarding_step`: `sphere → business → ig_wait | ig_connect → knowledge → finish → done`.
   Каждый переход — `PATCH /api/miniapp/tenant { onboardingStep }` (роут расширить новым enum);
   выход и повторный вход возвращают на незавершённый шаг (данные шагов подтягиваются из API:
   tenant + `GET /api/miniapp/onboarding/ig-account`).
3. **Язык онбординга = язык Telegram.** Клиент уже определяет локаль из языка Telegram
   (`resolveLocale(language_code)` в `TmaProvider`), но в БД `tenants.ui_locale` остаётся
   дефолтным `'de'`. При старте онбординга (первый экран, `onboarding_step` пуст/начальный)
   синхронизировать определённую локаль в тенанта: `PATCH /api/miniapp/tenant { uiLocale }`
   (расширить роут; Zod-enum по `Locale`). Все экраны онбординга, шаблон базы знаний и
   последующие сообщения бота идут на этой локали; поменять язык позже можно в настройках
   (T-028), это поведение не трогать.

## Экраны

4. **Экран 1 — сфера деятельности:** карточки-строки из `BUSINESS_SPHERES` (T-035): иконка +
   название (i18n), выбор строго одной (выделение рамкой/фоном в стиле темы Telegram).
   «Далее» активна после выбора → `PATCH tenant { businessSphere }` → экран 2.
5. **Экран 2 — данные бизнеса:** поле «Название бизнеса или ваше имя» (min 2 символа) и поле
   «Ваш Instagram-аккаунт» (плейсхолдер `@username`; live-валидация через `normalizeIgUsername`).
   «Далее» активна при валидных полях → `PATCH tenant { orgName }` +
   `POST /api/miniapp/onboarding/ig-account`:
   - `{ status: 'pending' }` → шаг `ig_wait` (экран 3а);
   - `{ status: 'approved' }` → шаг `ig_connect` (экран 3б);
   - 409 `taken` → инлайн-ошибка под полем, на экране 2 остаёмся.
6. **Экран 3а — заявка подана** (`ig_wait`): иконка-статус + текст: «Заявка на подключение
   Instagram подана. Подключение подтверждает администратор — это займёт немного времени.
   А пока перейдите к настройке описания вашего бизнеса». «Далее» активна всегда → шаг
   `knowledge`. (Сообщение от администратора с кнопкой подключения придёт в Telegram — T-037.)
7. **Экран 3б — подключение Instagram** (`ig_connect`): выполняется реальное подключение —
   OAuth-блок Business Login, как на `/app/connect-instagram`: кнопка получает URL из
   `/api/miniapp/ig/login-url`, открывает через `openLink`, статус перезапрашивается по фокусу
   и кнопке «Проверить». Вынеси этот блок из `/app/connect-instagram/page.tsx` в общий компонент
   `src/components/miniapp/InstagramConnectPanel.tsx` и используй в обоих местах (страница
   connect-instagram поведения не меняет). «Далее» активируется, когда статус подключения
   `active` → шаг `knowledge`.
8. **Экран 4 — шаблон базы знаний** (`knowledge`): заголовок «Опишите ваш бизнес», подсказка:
   «Мы подготовили шаблон для вашей сферы — замените примерные данные (адрес, часы, цены)
   на свои». Textarea (моноширинная, авто-высота), предзаполненная
   `getKbTemplate(businessSphere, locale)` — шаблон берётся на текущей локали интерфейса
   (см. шаг 3: она определена из языка Telegram и синхронизирована в `tenants.ui_locale`);
   при возврате на шаг — текущее значение `tenants.knowledge_base`, если оно уже сохранено.
   Кнопка внизу — «Завершить» (вместо «Далее»); активна при непустом тексте ≤ 20 000 символов →
   `POST /api/miniapp/onboarding/complete` → шаг `finish`.
9. **Экран 5 — финал** (`finish`): «Готово! Что дальше» — краткая шпаргалка по системе списком
   (иконка + 1–2 строки на пункт):
   - **Тест-чат (симулятор):** проверьте базу знаний — напишите боту как клиент и посмотрите
     черновики ответов;
   - **Входящие из Instagram:** когда клиент напишет в Direct, в этот Telegram-чат придёт
     карточка с черновиком — отправка одним нажатием «✅ Отправить»;
   - **Категории:** настройте категории обращений и правила ответа для каждой;
   - **Настройки:** база знаний и стиль ответов редактируются в любой момент;
   - **Дашборд:** статистика диалогов и статус подключения Instagram (+ если заявка ещё
     ожидает подтверждения — напоминание, что придёт сообщение от бота).
   Кнопка внизу — «Начать работу» → `PATCH tenant { onboardingStep: 'done' }` → редирект на
   дашборд (навигация снова видна).

## Бэкенд финализации

10. `app/api/miniapp/onboarding/complete/route.ts` POST (requireTenant), Zod
   `{ knowledgeBase: string }` (1…20 000 символов): проверить, что у тенанта заполнены
   `org_name` и `business_sphere` (иначе 409 `{ code: 'incomplete' }`); собрать
   `system_prompt = defaultSystemPrompt(orgName)` — НОВАЯ функция-шаблон в
   `src/lib/llm/prompts.ts` с фиксированным текстом (те же требования, что раньше запрашивались
   у LLM: представляться от имени {orgName}, отвечать кратко и дружелюбно на языке клиента,
   опираться только на базу знаний, не выдумывать цены и обещания, при неуверенности предложить
   дождаться ответа сотрудника); сохранить `knowledge_base` + `system_prompt`; вызвать rpc
   `seed_default_labels(tenantId)` (идемпотентно); `onboarding_step = 'finish'`. Повторный вызов
   допустим (перезаписывает КБ — пользователь мог вернуться назад).
11. i18n ru/de для всех строк экранов; HTML-эскейпинг не требуется (React), но тексты из
    шаблона нигде не рендерить как HTML.
12. Manual-test `docs/manual-tests/T-038.md`: два сквозных прохода (username не в реестре →
    ветка 3а; заранее подтверждённый username → ветка 3б с реальным OAuth), проверка
    неактивности «Далее» на каждом экране, выход/возврат на каждом шаге, кнопка из
    сообщения администратора ведёт на подключение, после финала навигация видна и открыт
    дашборд. Проверка языка: Telegram на русском → онбординг и шаблон на русском,
    Telegram на немецком (или любом другом) → на немецком; `tenants.ui_locale` после старта
    онбординга соответствует языку Telegram.
