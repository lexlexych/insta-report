# T-046 · Zernio: миграция БД (zernio_accounts, provider у drafts) и репозиторий

> **Перед началом:** прочитай `docs/research/zernio.md` §8 и `docs/plan.md` §5.
> Зависимости: T-004 (done). Эпик 12 «Instagram через Zernio», ветка эпика — `zernio`.

## Цель
Хранилище связки «тенант ↔ Zernio profile ↔ подключённый аккаунт» и признак провайдера
у черновиков. Старые таблицы (`ig_connections`, `ig_accounts`) НЕ меняются — прямая
Meta-интеграция продолжает работать как есть.

## Шаги выполнения
1. **Миграция** `supabase/migrations/0013_zernio.sql`:
   ```sql
   create table zernio_accounts (
     id uuid primary key default gen_random_uuid(),
     tenant_id uuid not null references tenants(id) on delete cascade,
     platform text not null default 'instagram' check (platform in ('instagram')),
     zernio_profile_id text not null,
     zernio_account_id text,
     username text,
     status text not null default 'pending'
       check (status in ('pending', 'active', 'disconnected', 'error')),
     connected_at timestamptz,
     disconnect_reason text,
     created_at timestamptz not null default now()
   );
   -- один аккаунт данного канала на тенанта (расширение на другие каналы = новые
   -- значения в check platform, без изменения схемы)
   create unique index zernio_accounts_tenant_platform_uniq
     on zernio_accounts (tenant_id, platform);
   -- маппинг вебхуков: account.accountId → тенант
   create unique index zernio_accounts_account_id_uniq
     on zernio_accounts (zernio_account_id) where zernio_account_id is not null;
   create index zernio_accounts_profile_idx on zernio_accounts (zernio_profile_id);

   alter table drafts add column provider text not null default 'meta'
     check (provider in ('meta', 'zernio'));
   alter table drafts add column zernio_conversation_id text;
   ```
   RLS: deny-all как у остальных таблиц (по образцу `0011_ig_accounts.sql`) — доступ только
   service-role. Применение миграции — шаг человека (`pnpm db:push`), см. OVERRIDES §3.
2. **`src/lib/db/types.gen.ts`** — дополнить вручную (файл пока ручной, см. шапку файла):
   таблица `zernio_accounts`, новые колонки `drafts.provider`, `drafts.zernio_conversation_id`.
3. **Репозиторий** `src/lib/db/zernioAccounts.ts` (реэкспорт в `src/lib/db/index.ts`):
   - `getForTenant(tenantId, platform = 'instagram'): Promise<ZernioAccountRow | null>`;
   - `ensureProfile(tenantId, platform, createProfileFn): Promise<string>` — вернуть
     `zernio_profile_id` существующей записи или создать запись `status='pending'`
     c профилем, полученным из колбэка (сам вызов Zernio API остаётся в роуте T-047 —
     репозиторий БД не должен ходить в сеть; допустима и упрощённая пара
     `insertPending(tenantId, platform, zernioProfileId)` + `getForTenant`);
   - `activate(tenantId, platform, { zernioAccountId, username }): Promise<void>` —
     `status='active'`, `connected_at=now()`, `disconnect_reason=null`; идемпотентно;
   - `getByZernioAccountId(zernioAccountId): Promise<ZernioAccountRow | null>`;
   - `getByZernioProfileId(zernioProfileId): Promise<ZernioAccountRow | null>`;
   - `setStatus(tenantId, platform, status, reason?): Promise<void>`;
   - `disconnect(tenantId, platform, reason): Promise<void>` — `status='disconnected'`,
     `disconnect_reason=reason` (запись НЕ удалять: сохраняем profile_id для переподключения).
   Ошибки — через `throwDb`/`isSupabaseCode` как в соседних репозиториях.
4. **`drafts`-репозиторий** (`src/lib/db/drafts.ts`): расширить `insertPending` полями
   `provider` (дефолт `'meta'`) и `zernio_conversation_id` (опционально); существующие
   вызовы не менять (дефолт сохраняет текущее поведение).
5. **Каскад удаления тенанта**: проверь `deleteCascade` в `src/lib/db/tenants.ts` (T-028) —
   `zernio_accounts` удаляется FK-каскадом; если там есть явный список таблиц, дополни.
6. **Краевые случаи**: повторный `activate` тем же аккаунтом (идемпотентность); `activate`
   при `zernio_account_id`, уже занятом другим тенантом (unique violation → типизированная
   ошибка, не «полотно» Supabase); `getByZernioAccountId(null/пустая строка)` → null.
7. **Demo-скрипт** `scripts/manual/T-046-demo.ts`: против dev-БД создать тестового тенанта,
   пройти цикл insertPending → activate → getByZernioAccountId → disconnect → печать строк,
   удалить тестовые данные.
8. **Manual-test** `docs/manual-tests/T-046.md`: чек-лист «что должно появиться после
   `pnpm db:push`» (таблица, 3 индекса, 2 колонки drafts, RLS) — по образцу T-002/T-036;
   прогон demo-скрипта.

## Критерии приёмки
- [ ] Миграция применяется на чистый dev-проект без ошибок; старые таблицы не тронуты
- [ ] Уникальность: один zernio-аккаунт на (tenant, platform); один тенант на zernio_account_id
- [ ] Репозиторий — единственная точка доступа к таблице; типизированные ошибки
- [ ] `drafts.provider` дефолтится в 'meta' — существующий пайплайн не требует правок
- [ ] Удаление тенанта каскадно удаляет его zernio_accounts
- [ ] lint, typecheck зелёные; manual-test с чек-листом схемы написан
