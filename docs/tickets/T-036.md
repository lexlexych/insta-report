# T-036 · БД: реестр Instagram-аккаунтов (ig_accounts) и сфера деятельности тенанта

> **Перед началом:** прочитай `docs/plan.md` §2.4, §5 (ревизия v1.2) и `docs/tickets/OVERRIDES.md`.
> Зависимости: T-034 (миграции продолжаются после `0010_remove_own_app.sql`).
> Контекст: для пилота Instagram-аккаунты добавляются в наше Meta App вручную; приложению нужен
> реестр таких аккаунтов со статусом подтверждения администратором.

## Цель

Миграция: таблица `ig_accounts` + колонка `tenants.business_sphere`; репозиторий
`src/lib/db/igAccounts.ts` с нормализацией username.

## Шаги выполнения

1. Миграция `supabase/migrations/0011_ig_accounts.sql`:
   - таблица `ig_accounts`:
     `id uuid pk default gen_random_uuid()`,
     `ig_username text not null unique` (хранится нормализованным: lowercase, без `@`),
     `tenant_id uuid null references tenants(id) on delete set null`,
     `status text not null default 'pending' check (status in ('pending','approved'))`,
     `requested_at timestamptz not null default now()`,
     `approved_at timestamptz null`,
     `approved_by_tg_id bigint null`;
   - `alter table tenants add column business_sphere text;` (nullable; значения — id сфер из
     T-035, enum на уровне приложения, не БД);
   - RLS: enable на `ig_accounts` без политик (deny-all для anon — как у остальных таблиц).
2. Обновить `src/lib/db/types.gen.ts` вручную по формату существующих таблиц
   (Row/Insert/Update для `ig_accounts`; `business_sphere` в `tenants`).
3. `src/lib/db/igAccounts.ts` (единственная точка доступа к таблице):
   - `normalizeIgUsername(input: string): string | null` — trim, срез ведущего `@`, lowercase,
     валидация `^[a-z0-9._]{1,30}$`; невалидный ввод → null. Экспортировать: нужен и в API, и в UI;
   - `findByUsername(username)` — по нормализованному значению;
   - `createPending(username, tenantId)` — insert со статусом `pending`;
   - `approve(id, adminTgId)` — атомарный `UPDATE ... WHERE status = 'pending' RETURNING`
     (повторное подтверждение → null, по образцу `claimPendingToSending` в drafts);
   - `getByTenant(tenantId)` — последняя заявка тенанта;
   - `bindTenant(id, tenantId)` — привязка approved-записи без владельца к тенанту.
4. Репозиторий `tenants`: поддержать `business_sphere` в patch-методе обновления.
5. Manual-test `docs/manual-tests/T-036.md`: применение `pnpm db:push` человеком на dev-проекте,
   чек-лист «какие таблицы/колонки/индексы должны появиться» (по образцу T-002 в OVERRIDES),
   при необходимости demo-скрипт `scripts/manual/T-036-demo.ts` для прогона репозитория.

## Критерии приёмки

- [ ] Миграция применяется на dev-проект после 0010 без ошибок; `types.gen.ts` соответствует схеме
