# T-026 · UI тест-чата

> **Перед началом:** прочитай `docs/plan.md` §1, §5. Зависимости: T-006, T-025.

## Цель
Страница `/app/simulator` — интерфейс мессенджера для проверки ассистента «глазами клиента».

## Шаги выполнения
1. Состояние: `messages: {role, text, label?}[]` в React state (+ sessionStorage, чтобы переживать случайный свайп навигации).
2. Разметка: скролл-лента пузырей (client — справа, акцентный цвет; assistant — слева, secondary-bg, сверху бейдж 🏷 {label}); авто-скролл вниз при новом сообщении; typing-индикатор (три точки) на время запроса.
3. Ввод: textarea авто-рост до 4 строк + кнопка отправки; Enter = отправка (Shift+Enter — перенос); disabled во время запроса.
4. Отправка: push client-сообщения → POST simulator/message со всей историей → push assistant-сообщения c label. Ошибки: 429 daily_limit → системный пузырь «Дневной лимит тест-чата исчерпан, продолжите завтра»; 409 onboarding_required → экран-заглушка c кнопкой «Пройти онбординг»; прочее → пузырь «Ошибка, попробуйте ещё раз» + кнопка повтора последнего сообщения.
5. Шапка: подсказка-баннер (закрываемый, флаг в localStorage): «Вы пишете как клиент. Ответы — черновики ассистента; в реальной работе они приходят вам в Telegram на подтверждение». Кнопка «⟲ Начать заново» (confirm → очистка state).
6. Первый вход (пустая история): 3 чипа c примерами вопросов («Сколько стоит…?», «Какой у вас адрес?», «Можно записаться на завтра?») — тап подставляет в input.
7. Компонентные тесты: отправка добавляет оба пузыря (мок fetch); 429 показывает лимит-сообщение; reset очищает; контекст (вся история) уходит в запрос.

## Критерии приёмки
- [ ] Диалог 5+ сообщений сохраняет контекст (история уходит целиком)
- [ ] Все ошибки бэкенда обработаны нефатально
- [ ] UX-подсказка объясняет связь c реальным потоком
