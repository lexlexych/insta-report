# T-049 · Zernio: входящие сообщения → существующий пайплайн (парсинг, контекст, черновик)

> **Перед началом:** прочитай `docs/research/zernio.md` §6–8 и `docs/plan.md` §5. Изучи
> пайплайн: `src/lib/pipeline/{parse,types,handleIgEvent,handleIncoming,handleEcho,context,key}.ts`.
> Зависимости: T-046, T-048 (плюс done: T-016, T-017). Эпик 12, ветка эпика — `zernio`.

## Цель
Входящий DM, пришедший вебхуком Zernio, проходит ту же цепочку, что и прямой Meta-вебхук:
нормализация → дедуп → контекст → классификация → черновик → карточка в Telegram. Логика
классификации/генерации/доставки НЕ дублируется — переиспользуется через параметры.

## Архитектурное решение
Пайплайн уже написан с dependency-injection (`deps` в `handleIncoming`/`buildContext`).
Zernio-путь не трогает Meta-путь: свой маппер событий + свой сборщик контекста, дальше —
общие `classify`/`generateDraft`/`deliverDraft`.

## Шаги выполнения
1. **Маппер** `src/lib/zernio/mapEvent.ts`:
   `mapZernioMessage(payload): ZernioPipelineEvent | null` — из payload `message.received`
   / `message.sent` (форма — research §7) собрать объект, СОВМЕСТИМЫЙ с `IgEvent`
   (`src/lib/pipeline/types.ts`), плюс zernio-поля:
   - `kind`: `direction === 'incoming'` → `'incoming'`, `'outgoing'` → `'echo'`;
   - `accountId` = `account.accountId` (zernio account id — стабильный ID «нашей стороны»);
   - `contactId` = `conversation.participantId` (IGSID клиента; фолбэк для incoming —
     `message.sender.id`);
   - `text` = `message.text ?? ''`; `hasAttachments` / `attachmentTypes` — из
     `message.attachments` (type отсутствует → `'unknown'`, как в `parse.ts`);
   - `mid` = `message.platformMessageId` (**mid Meta** — единый namespace с прямой
     интеграцией, это даёт кросс-провайдерный дедуп, research §8.1);
   - `ts` = `Date.parse(message.sentAt)`;
   - экстра: `provider: 'zernio'`, `zernioConversationId: message.conversationId`,
     `zernioAccountId: account.accountId`, `contactUsername:
     conversation.participantUsername ?? message.sender.username ?? null`;
   - `platform !== 'instagram'` или нет `platformMessageId` → `null` (debug-лог).
   `conversationKey` получается стандартным `conversationKey(ev)` =
   `'{accountId}:{contactId}'` — для zernio это `'{zernio_account_id}:{IGSID}'`.
   Ключи Meta- и Zernio-переписок НЕ совпадают — это сознательно: непрерывность истории
   при смене провайдера не требуется, зато нет коллизий.
2. **Обработчик** `src/lib/zernio/handleMessage.ts`, `handleZernioMessage(payload, deps?)`
   (его вызывает роут T-048; убрать заглушку `TODO(T-049)`):
   - `mapZernioMessage` → null → выход;
   - тенант: `zernioAccounts.getByZernioAccountId(ev.zernioAccountId)`; нет записи или
     `status !== 'active'` → debug-лог, выход;
   - дедуп: `processedEvents.tryInsert(tenantId, ev.mid)` → false → выход (гасит и ретраи
     Zernio, и двойную доставку Meta-директ+Zernio на одном IG-аккаунте, и echo собственных
     отправок T-050);
   - `kind === 'echo'` → существующий `handleEcho(tenantId, ev)` — он оперирует только
     `conversationKey`/drafts/messageLog, к IG API не ходит, работает как есть;
   - фильтры «пустое сообщение», «чистый репост» — та же логика, что в `handleIgEvent`
     (вынеси в общий хелпер в `src/lib/pipeline/`, НЕ копипасть);
   - incoming → `handleIncoming(tenantId, ev, zernioDeps)` (п.3).
3. **Интеграция с `handleIncoming`** (`src/lib/pipeline/handleIncoming.ts`) — минимальный
   рефакторинг, поведение Meta-пути бит-в-бит прежнее:
   - гейт «есть активный `ig_connections`» и сборка контекста через Graph API — это
     dep-функции; для zernio-пути передай deps: `getConnection` → активная запись
     `zernio_accounts` (адаптер с полями, которые реально использует пайплайн),
     `buildContext` → zernio-вариант (п.4);
   - если текущая сигнатура не позволяет подменить какой-то шаг — расширь `deps`
     (с дефолтами = текущее поведение), но НЕ меняй Meta-ветку по существу;
   - `deliverDraft` получает `provider: 'zernio'` и `zernio_conversation_id` и пишет их
     в `drafts.insertPending` (колонки из T-046); для Meta-пути — дефолт `'meta'`/null.
4. **Контекст** `src/lib/zernio/context.ts`, `buildZernioContext(account, ev, deps?)`:
   - история: `getConversationMessages(ev.zernioConversationId, account.zernio_account_id,
     { limit: 20, sortOrder: 'desc' })` → привести к `ConversationMessage { text, fromId,
     createdTime }`: `fromId` = `senderId`, но проще и надёжнее направление:
     `direction === 'outgoing'` → «Бизнес», иначе «Клиент» — переиспользуй `deriveContext`
     из `src/lib/pipeline/context.ts`, подав `fromId = direction === 'outgoing' ?
     igAccountIdПлейсхолдер : senderId` (посмотри фактическую сигнатуру `deriveContext`
     и передай то, что она сравнивает; при необходимости экспортируй из context.ts
     небольшой перегруженный хелпер);
   - username клиента: `ev.contactUsername` (сетевой вызов не нужен); пустая история →
     фолбэк на само `ev` (как в `buildContext`);
   - ошибки Zernio API (таймаут, 429) → те же правила деградации, что у Meta-пути
     (посмотри обработку ошибок `getConversation` в `context.ts` и повтори).
5. **Краевые случаи**: сообщение без текста с вложением (карточка «[вложение]» — паритет
   с Meta-путём); `message.sent` нашей же отправки T-050 (уже в processed_events → тихий
   выход); вебхук по аккаунту в статусе `disconnected`; `sentAt` не парсится → `ts = Date.now()`;
   квитанции `message.read`/`message.delivered` на этот endpoint не подписаны, но если
   придут — маппер вернёт null по отсутствию `platformMessageId`/`text`+`attachments`.
6. **Demo-скрипт** `scripts/manual/T-049-demo.ts`: подать в `handleZernioMessage` фикстуры
   payload'ов (incoming текст, incoming вложение, outgoing/echo, дубликат mid, чужой
   accountId) с замоканными deps (без сети и БД — по образцу `T-016-demo.ts`/`T-017-demo.ts`),
   печатать решения пайплайна.
7. **Manual-test** `docs/manual-tests/T-049.md`: подготовка — подключённый через Zernio
   IG-аккаунт (T-047/T-048), туннель; проверки — DM с постороннего IG-аккаунта → карточка
   в Telegram с классификацией и черновиком (как в Meta-флоу); повторная доставка вебхука →
   второй карточки нет; ответ владельца вручную из приложения Instagram → pending-карточка
   отменяется (echo; research §9.2 — если `message.sent` для ручных ответов не приходит,
   зафиксировать факт в manual-test и notes тикета); DM при одновременно активном
   Meta-директ подключении того же аккаунта → карточка одна (дедуп по mid).

## Вне объёма
- Отправка ответа через Zernio и регистрация собственных mid — T-050.
- UI-статусы — T-051.

## Критерии приёмки
- [ ] DM через Zernio порождает карточку в Telegram по той же цепочке (метка, черновик,
      топик) без дублирования кода classify/draft/deliver
- [ ] Дедуп по `platformMessageId` гасит ретраи Zernio и двойную доставку при параллельном
      Meta-подключении
- [ ] Echo (outgoing) отменяет pending-карточку и пишет `manual` в message_log
- [ ] Драфт создаётся с `provider='zernio'` и заполненным `zernio_conversation_id`
- [ ] Meta-путь не изменён поведенчески (diff `handleIncoming`/`context.ts` — только
      расширение deps с дефолтами)
- [ ] lint, typecheck зелёные; demo-скрипт и manual-test написаны
