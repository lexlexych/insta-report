# T-024 ·  Топики в личном чате бота по категориям

> **Перед началом:** прочитай `docs/plan.md` §2.3, §5, `docs/tickets/OVERRIDES.md` §1. Зависимости: T-021, T-023. Выполнять после стабилизации MVP.
> **Заменяет** прежнюю формулировку про отдельную форум-группу: **отдельной группы не будет**. Категории раскладываются по топикам прямо в личном чате владельца с ботом — Telegram разрешил темы (topics) в приватных чатах с ботом (**Bot API 9.3, 31.12.2025**).

## Цель
Черновики и лог отправленных ответов раскладываются по топикам прямо в личном чате владельца с ботом: топик = категория (метка). При создании категории в Mini App автоматически создаётся соответствующий топик, и все сообщения этой категории уходят в его `message_thread_id`. Если у пользователя темы в чате выключены или любой вызов топиков падает — поведение бесшовно деградирует до обычного (плоского) чата **без потери черновиков**.

## Контекст: функция Telegram
- **Bot API 9.3 (31.12.2025)** разрешил ботам работать с топиками не только в супергруппах, но и в **приватных чатах** с ботом.
- Владелец бота один раз включает **«Threaded Mode»** (темы в приватных чатах) через **@BotFather** — операционный шаг, не код.
- `User.has_topics_enabled` — включён ли режим тем в приватном чате с этим пользователем. При `false` — работаем в плоском режиме.
- В приватном чате бот создаёт топики методом `createForumTopic` **без прав администратора** (в отличие от супергрупп).
- `message_thread_id` поддержан в приватных чатах в `sendMessage`/`sendPhoto`/… (в проекте **уже** прокинут в `sendMessageHTML`). Пропуск `message_thread_id` = General-топик / плоский чат.
- `editForumTopic` / `deleteForumTopic` также работают в приватных чатах.

## Что уже есть в коде (переиспользуем)
- `labels.tg_thread_id bigint null` — **уже есть** (миграция `0001`). Миграция под него не нужна.
- `tenants.tg_chat_id bigint` — **уже есть** (`0001`). Это и есть `chatId` личного чата, где создаём топики.
- `src/lib/tg/api.ts`: `sendMessageHTML(chatId, html, keyboard?, threadId?)` уже кладёт `message_thread_id` при заданном `threadId`.
- `src/lib/pipeline/deliver.ts`: `deliverDraft` уже шлёт в `tenant.tg_chat_id` — надо лишь прокинуть `threadId` метки.
- CRUD категорий из T-023 (`src/lib/db/labels.ts` + `/api/miniapp/labels`) — точка, куда вешаем создание/переименование/удаление топика.
- Миграции в проекте: `0001, 0002, 0003, 0006, 0007` → следующий свободный номер — **`0008`**.

## Шаги выполнения
1. **Миграция `0008` (минимальная):** `tenants.tg_topics_enabled boolean not null default false` (кэш `has_topics_enabled`, чтобы не дёргать `getChat` на каждую доставку); `tenants.history_thread_id bigint null` (топик «История»). `labels.tg_thread_id` уже существует — НИЧЕГО не добавлять. Регенерировать `src/lib/db/types.gen.ts`; применяет миграцию человек через `pnpm db:push` (OVERRIDES §3).
2. **Обёртки топиков в `src/lib/tg/api.ts`:** `createForumTopic(chatId, name): Promise<number>` (вернуть `message_thread_id` из `ForumTopic`), `editForumTopic(chatId, threadId, name)`, `deleteForumTopic(chatId, threadId)`, `readTopicsEnabled(chatId): Promise<boolean>` (через `getChat().has_topics_enabled` или из уже приходящих апдейтов с `User`). Обёртки тонкие, ошибки пробрасывают наверх.
3. **`src/lib/tg/topics.ts` (новый) — идемпотентные ensure-хелперы:** `ensureLabelTopic(tenant, label): Promise<number|null>` — если `tg_topics_enabled=false` → `null` (плоский режим); если `label.tg_thread_id` есть → вернуть; иначе `createForumTopic(tenant.tg_chat_id, label.name)` → сохранить `tg_thread_id` → вернуть; любая ошибка → лог + `null`. `ensureHistoryTopic(tenant)` — аналогично для «📜 История» (`tenants.history_thread_id`). `syncTopicsEnabled(tenant, valueFromUpdate?)` — обновляет кэш `tg_topics_enabled`. **Повторный вызов при заполненном `tg_thread_id` НЕ дёргает `createForumTopic`.**
4. **Автосоздание топика при создании категории (главное требование):** в `POST /api/miniapp/labels` после вставки метки, если `tenant.tg_topics_enabled` — вызвать `ensureLabelTopic` и сохранить `tg_thread_id`. Ошибка создания топика НЕ ломает создание категории (метка создаётся, `tg_thread_id=null`, топик до-создастся лениво при первой доставке). `PUT /labels/[id]` (переименование): если `tg_thread_id` есть и темы включены → `editForumTopic(name)` (ошибку глотаем, имя в БД первично). `DELETE /labels/[id]` (nice-to-have): по возможности `deleteForumTopic`, ошибку глотаем.
5. **Доставка черновиков по топикам — `deliver.ts`:** в `deliverDraft` `threadId = await ensureLabelTopic(tenant, label)` для метки черновика, передать в `sendMessageHTML`. Черновик без метки / метка без топика / темы выключены → `threadId=null` → General-топик (плоско), как сейчас. Ошибка отправки именно из-за топика (тема удалена/недоступна) → повтор отправки **без** `message_thread_id` → черновик НЕ теряется; владельца не спамить (лог достаточно).
6. **Лог отправленных в топик «История» — интеграция с T-021:** после успешной отправки ответа (кнопка «✅ Отправить») `threadId = ensureHistoryTopic(tenant)`; сообщение вида «`@{username} · {label}`\n`Вопрос: {pendingText}`\n`Ответ: {sentText}`». Переиспользовать `message_log` (`direction='out'/'manual'`). Ошибка истории не влияет на факт отправки ответа в Instagram.
7. **Индикатор в Mini App (лёгкий, вместо старого radio):** в настройках доставки — статус «Темы в чате: включены / выключены» на основе `tenants.tg_topics_enabled` + подсказка «Чтобы черновики раскладывались по категориям, включите Threaded Mode для бота в @BotFather». Никаких radio «Личные / Группа» и инструкций про группу — их больше нет.
8. **Операционный шаг (задокументировать, не код):** в `docs/deploy.md` и `docs/manual-tests/T-024.md` зафиксировать разовое действие: @BotFather → включить «Threaded Mode» для бота. Без него `has_topics_enabled=false` у всех и работает плоский режим — это ожидаемо и валидно.

## Краевые случаи (учесть в коде — по OVERRIDES §1, вместо автотестов)
- Топик создаётся один раз на метку (повторный `ensureLabelTopic` не создаёт новый).
- `has_topics_enabled=false` → доставка плоская, `createForumTopic` не вызывается.
- `createForumTopic` упал → категория/доставка не ломаются, fallback в General.
- Пользователь вручную удалил топик → следующая доставка не падает (пере-создание или General).
- Переименование метки синхронизирует имя топика; метка без `tg_thread_id` до-создаёт топик лениво.
- Лог «История» падает → факт отправки ответа в Instagram не затронут.

## Вне охвата
- Отдельная супергруппа / форум-группа и любой групповой режим доставки.
- Захват `chat_id` через `my_chat_member`.
- Управление правом пользователя создавать топики (`allows_users_to_create_topics`) — топики создаёт сам бот.
- Автотесты (OVERRIDES §1).

## Ручная проверка
`docs/manual-tests/T-024.md` — прогон: (1) BotFather Threaded Mode вкл; (2) создать категорию → появился топик; (3) переименовать → имя топика обновилось; (4) доставка черновика метки → карточка в нужном топике; (5) вторая доставка той же метки → новый топик не создан; (6) отправка ответа → запись в «История»; (7) выключить темы / удалить топик → доставка не падает, уходит в General.

## Критерии приёмки
- [ ] Режим топиков полностью опционален: при `has_topics_enabled=false` поведение идентично текущему (плоский личный чат), без регрессов доставки.
- [ ] Создание категории при включённых темах создаёт ровно ОДИН топик; переименование обновляет имя топика.
- [ ] Идемпотентность: вторая доставка черновика той же метки НЕ вызывает `createForumTopic`.
- [ ] Любая ошибка топиков деградирует до General-топика **без потери черновика**; владелец не заваливается уведомлениями.
- [ ] Отправленные ответы логируются в топик «📜 История».
- [ ] Отдельной группы в решении нет; `group_chat_id` и `delivery_mode` не вводятся.
- [ ] Definition of Done: `pnpm lint && pnpm typecheck` (без `pnpm test`, OVERRIDES §1).
