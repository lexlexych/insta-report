# T-037 · Заявки на подключение Instagram: проверка, уведомление и подтверждение администратором

> **Перед началом:** прочитай `docs/plan.md` §1, §2, §5 (ревизия v1.2) и `docs/tickets/OVERRIDES.md`.
> Зависимости: T-005 (бот), T-033 (deep-link `?startapp=connect`), T-036 (ig_accounts).
> Контекст: пользователь на онбординге вводит свой Instagram-аккаунт; если его ещё нет в нашем
> Meta App — создаётся заявка, администратор добавляет аккаунт в Meta App вручную и подтверждает
> кнопкой в Telegram, после чего пользователь получает приглашение подключиться.

## Цель

Бэкенд-цикл заявки: эндпоинт проверки/создания заявки → уведомление администраторам с inline-кнопкой
«Подтвердить» → смена статуса → уведомление пользователя со ссылкой на экран подключения.

## Шаги выполнения

1. `app/api/miniapp/onboarding/ig-account/route.ts` (requireTenant):
   - **POST**, Zod-body `{ igUsername: string }` → `normalizeIgUsername`; невалидный → 400
     `{ code: 'invalid_username' }`. Дальше по `findByUsername`:
     - записи нет → `createPending(username, tenantId)` → уведомить админов (п.2) →
       200 `{ status: 'pending' }`;
     - есть `pending` этого же тенанта → 200 `{ status: 'pending' }` (идемпотентно, админов
       повторно НЕ уведомлять);
     - есть `approved` с `tenant_id = null` или этого тенанта → `bindTenant` при необходимости →
       200 `{ status: 'approved' }`;
     - есть запись другого тенанта (любой статус) → 409 `{ code: 'taken' }` (фронт покажет
       «аккаунт уже используется другим пользователем, напишите в поддержку»).
   - **GET** → `{ status: 'none' | 'pending' | 'approved', igUsername? }` по `getByTenant`
     (для восстановления состояния онбординга и поллинга экрана ожидания).
2. Уведомление администраторам (`src/lib/tg`): каждому id из `env.ADMIN_TELEGRAM_IDS`
   сообщение «🆕 Заявка на подключение Instagram: @{username}» + название бизнеса
   (`tenants.org_name`, если уже заполнено) + `telegram_user_id` заявителя; inline-кнопка
   «✅ Подтвердить» c `callback_data = "igacc_ok:{id}"` (uuid, укладывается в 64 байта).
   Ошибка доставки одному админу не роняет запрос и не мешает остальным (ошибки логировать).
3. Обработчик callback `igacc_ok:{id}` в grammY-боте:
   - `from.id` не входит в `ADMIN_TELEGRAM_IDS` → `answerCallbackQuery('Недостаточно прав')`, выход;
   - `igAccounts.approve(id, from.id)`; null (уже подтверждено/не найдено) →
     `answerCallbackQuery('Уже подтверждено')` + убрать клавиатуру у сообщения;
   - успех → отредактировать сообщение админа: «✅ Подтверждено: @{username}» (без клавиатуры);
     остальным админам сообщения не редактировать (достаточно идемпотентности approve);
   - уведомить пользователя (tenant по `tenant_id` заявки, чат `tg_chat_id`):
     «Ваш Instagram-аккаунт @{username} подтверждён — теперь можно подключиться» + inline-кнопка
     с URL `https://t.me/{TELEGRAM_BOT_USERNAME}?startapp=connect`. Если `tenant_id` пуст —
     только подтвердить админу.
4. Проверить обработку `start_param = 'connect'` в Mini App (появился в T-033): при открытии
   с этим параметром пользователь должен попадать на экран подключения Instagram
   (`/app/connect-instagram`; после T-038 — на шаг подключения онбординга, если онбординг
   не завершён). Если обработки нет — добавить в клиентскую auth-обвязку.
5. Никаких новых env: администраторы — `ADMIN_TELEGRAM_IDS`, username бота —
   `TELEGRAM_BOT_USERNAME` (оба уже в `src/lib/env.ts`).
6. Manual-test `docs/manual-tests/T-037.md`: полный цикл заявки (новый username → сообщение
   админу → «Подтвердить» → сообщение пользователю → кнопка ведёт в Mini App), повторная подача,
   повторное нажатие «Подтвердить», username чужого тенанта → 409, невалидный username → 400.

## Критерии приёмки

- [ ] Новый username создаёт `pending`-заявку и ровно одно уведомление каждому админу;
      повторный POST того же тенанта заявок и уведомлений не плодит
- [ ] «Подтвердить» работает только для админов, идемпотентен, переводит запись в `approved`
      и отправляет пользователю сообщение с кнопкой-ссылкой на экран подключения
- [ ] Username, занятый другим тенантом, → 409; невалидный → 400; сравнение — по
      нормализованной форме (`@FooBar` и `foobar` — один аккаунт)
- [ ] Секреты/токены в сообщениях и логах не фигурируют; тексты — через i18n там, где уходят
      пользователю (сообщения админам можно оставить на русском)
- [ ] `pnpm lint && pnpm typecheck` зелёные; manual-test написан
