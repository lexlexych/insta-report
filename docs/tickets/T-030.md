# T-030 · Rate limiting и защита эндпоинтов

> **Перед началом:** прочитай `docs/plan.md` §2.3, §5. Зависимости: T-011, T-025.

## Цель
Лимитер на Postgres, аудит защиты всех роутов, security-заголовки.

## Шаги выполнения
1. Миграция 0008: `rate_limits (key text, window_start timestamptz, count int, primary key (key, window_start))` + RLS enable.
2. `src/lib/rateLimit.ts` — фиксированные окна (проще sliding и достаточно):
   - `checkLimit(key: string, limit: number, windowSec: number): Promise<{ ok: boolean; remaining: number }>`;
   - реализация одним RPC (миграция 0008, функция `bump_rate_limit(p_key, p_window_start, p_limit)` → insert on conflict do update set count=count+1 returning count) — атомарно, без гонок;
   - ключи-конвенции: `miniapp:{tenantId}`, `sim:{tenantId}`, `wh:{tenantId}`, `alert:{kind}`.
3. Применение:
   - в apiHandler (T-007): для /api/miniapp/* после requireTenant — `checkLimit('miniapp:'+id, 60, 60)`; превышение → 429 `{ code:'rate_limited', retryAfterSec }`;
   - вебхук IG (T-011): `checkLimit('wh:'+tenantId, 300, 60)` — при превышении отвечай 200 (!) и дропай событие c логом warn (не 429: Meta начнёт ретраить и усилит шторм);
   - симулятор: переведи дневной лимит T-025 на `checkLimit('sim:'+id, 30, 86400)` и удали колонку-костыль, если делал(и);
   - alerts (T-029): переведи троттлинг на `alert:{kind}`.
4. Очистка: cron `/api/cron/cleanup` (schedule `0 5 * * *`, та же защита CRON_SECRET): delete rate_limits where window_start < now()-'2 days'; delete processed_events older 14 days. Добавь в vercel.json.
5. Заголовки: в `next.config.js` → headers() для `/app/*`: CSP (`default-src 'self'; connect-src 'self' https://api.telegram.org; frame-ancestors https://web.telegram.org https://*.telegram.org;` — подбери минимально рабочий для Mini App, проверь, что SDK и стили не ломаются), `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`.
6. **Аудит-тест** `tests/routes.audit.test.ts`: программно собери список файлов `app/api/**/route.ts`; для каждого проверь (импорт-анализ по исходнику), что присутствует один из маркеров защиты: `requireTenant` | `verifyIgSignature` (вынеси проверку подписи T-011 в именованную функцию) | `TELEGRAM_WEBHOOK_SECRET` | `CRON_SECRET`. Новый незащищённый роут валит тест.
7. Тесты лимитера: 60-й ок / 61-й fail в одном окне; новое окно сбрасывает; вебхук при переливе возвращает 200 и не вызывает handleIgEvent.

- [ ] Шторм вебхуков не роняет и не дороже лимита LLM-вызовов
