# T-025 · Бэкенд тест-чата (симулятор)

> **Перед началом:** прочитай `docs/plan.md` §1, §5. Зависимости: T-018, T-019. Критично: используются ТЕ ЖЕ функции пайплайна, не копии.

## Цель
`POST /api/miniapp/simulator/message` — прогон classify+draft по виртуальной беседе.

## Шаги выполнения
1. Роут (requireTenant): Zod body `{ messages: [{ role: 'client'|'assistant', text: string(1..1000) }] (1..40 элементов) }` — историю держит фронт, бэкенд stateless.
2. Лимит: 30 запросов/сутки на tenant. Реализация: `usage_stats` не подходит по семантике → добавь колонку `usage_stats.simulator_calls int default 0` (миграция 0006 + параметр в increment_usage) и проверяй `getRange(today)` перед вызовом; превышение → 429 `{ code: 'daily_limit' }`.
3. Преобразование входа в ConversationContext-совместимый вид: history-строки («Клиент:»/«Бизнес:» — assistant считаем бизнесом), pendingText = последние подряд client-сообщения (переиспользуй `deriveContext`: смапь messages в формат {text, fromId, createdTime} c fromId = 'sim-client'|igAccountId-заглушка 'sim-biz').
4. Вызовы: `labels.listByTenant` → `classify(...)` → `generateDraft(...)` (ev-заглушка `{ hasAttachments:false, text: pendingText }`) → increment simulator_calls → ответ `{ label: {id,name}, draft: string, usage }`.
5. Тенант без knowledge_base → 409 `{ code: 'onboarding_required' }`.
6. Тесты (spy на реальные функции classify/generateDraft c моком llm): вызываются именно они; лимит: 30-й ок, 31-й → 429; без kb → 409; деривация pendingText из messages корректна.

## Критерии приёмки
- [ ] Ноль дублирования пайплайн-логики (spy-тест подтверждает переиспользование)
- [ ] Дневной лимит работает и покрыт тестом
