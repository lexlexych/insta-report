# T-018 · Классификация обращения по меткам

> **Перед началом:** прочитай `docs/plan.md` §2.2, §5. Зависимости: T-008, T-017.

## Цель
`src/lib/pipeline/classify.ts`: выбор метки тенанта для входящего обращения.

## Шаги выполнения
1. Заполни `classifyPrompt(labels, history, pendingText)` в `src/lib/llm/prompts.ts`:
   - system: «Ты — классификатор обращений клиентов. Выбери РОВНО ОДНУ категорию из списка или верни пустую строку, если ни одна не подходит. Отвечай JSON {"label": "..."}»;
   - user: список категорий как `- {name}: {description}` + «История диалога:\n{history}» + «Новое обращение:\n{pendingText}».
2. `classify(tenant, labels, ctx): Promise<{ label: Label; usage }>`:
   - если у тенанта только «Без категории» → верни её без LLM-вызова (usage нулевой);
   - `completeJSON(MODEL_CLASSIFY, схема z.object({ label: z.string() }), tenantId)`;
   - матчинг: `labels.find(l => l.name.trim().toLowerCase() === answer.trim().toLowerCase())`; не найдено ИЛИ пустая строка → метка «Без категории» (найди по имени; отсутствует — первая по sort);
   - LlmError → залогируй и верни «Без категории» (классификация не должна ронять пайплайн).
3. Тесты `tests/pipeline.classify.test.ts` (мок llm):
   - точное имя → метка; имя в другом регистре/с пробелами → метка;
   - выдуманное имя / пустая строка → «Без категории»;
   - одна метка у тенанта → LLM не вызывался;
   - LlmError → «Без категории», исключение не пробрасывается.

## Критерии приёмки
- [ ] Классификация никогда не бросает исключение наружу
- [ ] Экономия вызова при единственной метке реализована
