# T-048 · Zernio: глобальный вебхук `/api/wh/zernio` и события аккаунта

> **Перед началом:** прочитай `docs/research/zernio.md` §7 (подпись, ретраи, payload'ы) и
> `docs/plan.md` §5. Посмотри Meta-аналог: `app/api/wh/ig/route.ts` + `src/lib/ig/webhook.ts`.
> Зависимости: T-045, T-046. Эпик 12 «Instagram через Zernio», ветка эпика — `zernio`.

## Цель
Единая точка приёма всех вебхуков Zernio (вебхук у Zernio глобальный на весь наш аккаунт,
не per-tenant): проверка подписи, быстрый ACK, маршрутизация по типу события и тенанту.
В этом тикете — каркас + события `account.*`; события `message.*` передаются в обработчик,
который реализует T-049 (здесь — заглушка `// TODO(T-049)`).

## Шаги выполнения
1. **Верификация подписи** `src/lib/zernio/webhook.ts`:
   `verifyZernioSignature(rawBody: string, header: string | null): boolean` —
   lowercase hex HMAC-SHA256(rawBody, `env.ZERNIO_WEBHOOK_SECRET`), сравнение через
   `crypto.timingSafeEqual` (по образцу `verifySignature` в `src/lib/ig/webhook.ts`;
   ВАЖНО: заголовок `X-Zernio-Signature` — чистый hex, БЕЗ префикса `sha256=`).
2. **Роут** `app/api/wh/zernio/route.ts`, только `POST`:
   - `!isZernioEnabled()` → 404;
   - читать **raw body** до JSON.parse (подпись считается от сырых байт);
   - невалидная/отсутствующая подпись → 401;
   - распарсить `{ id, event, timestamp, ... }` (zod-схема с `.passthrough()` по образцу
     `src/lib/pipeline/parse.ts`); не-JSON / без `event` → 200 + debug-лог (Zernio не должен
     ретраить мусор);
   - **ACK ≤ 5 секунд**: ответить 200 сразу, обработку — в `waitUntil(...)` с
     `logPipelineError`, как в `app/api/wh/ig/route.ts`.
3. **Маршрутизация** `src/lib/zernio/handleWebhook.ts`, `handleZernioWebhook(payload)`:
   - `account.connected` → `zernioAccounts.getByZernioProfileId(account.profileId)`;
     найдено → `activate(tenant_id, 'instagram', { zernioAccountId: account.accountId,
     username: account.username })` (идемпотентный дубль callback-а из T-047 — belt-and-braces,
     если пользователь закрыл браузер до redirect'а); не найдено → debug-лог, выход;
   - `account.disconnected` → `getByZernioAccountId(account.accountId)`; найдено →
     `disconnectionType === 'intentional'` ? `disconnect(..., reason)` :
     `setStatus(..., 'error', reason)` + Telegram-уведомление владельцу тенанта
     («Подключение Instagram через Zernio разорвано, переподключите») — по образцу алертов
     в `app/api/cron/refresh-tokens/route.ts`;
   - `message.received` / `message.sent` → `handleZernioMessage(payload)` из T-049;
     до T-049 — заглушка `// TODO(T-049)` c debug-логом;
   - неизвестный `event` → лог, 200.
   - Дедуп ретраев Zernio (at-least-once): для `account.*` идемпотентности операций
     достаточно; для `message.*` дедуп по `platformMessageId` делает T-049 через
     `processed_events`.
4. **Setup-скрипт** `scripts/manual/zernio-setup-webhook.ts` (человек запускает один раз
   при деплое): через клиент T-045 `listWebhookSettings()`; настройка с именем
   `insta-reply` есть → `updateWebhookSetting`, нет → `createWebhookSetting({
   name: 'insta-reply', url: env.APP_BASE_URL + '/api/wh/zernio',
   secret: env.ZERNIO_WEBHOOK_SECRET,
   events: ['message.received', 'message.sent', 'account.connected', 'account.disconnected'],
   isActive: true })`. Печать итоговой настройки без секрета. Помни: максимум 10 вебхуков
   на аккаунт Zernio; после 10 подряд ошибок доставки Zernio сам выключает вебхук
   (`isActive=false`) — скрипт при повторном запуске это чинит (обновляет isActive).
5. **Краевые случаи**: `account.connected` для профиля, у которого в БД уже `active` с
   ДРУГИМ `zernio_account_id` (пользователь переподключил другой IG-аккаунт — перезаписать
   поля); payload без `account.profileId` (спека допускает отсутствие — выйти по debug-логу);
   повторная доставка того же события (тот же `X-Zernio-Event-Id`) — операции идемпотентны.
6. **Manual-test** `docs/manual-tests/T-048.md`: подготовка — env, туннель, запуск
   setup-скрипта; проверки — `POST /v1/webhooks/test` из дашборда Zernio (или `curl` с
   самодельной подписью) → 200; подключение аккаунта (T-047) → в логах `account.connected`
   и строка активирована; негативные — запрос без подписи → 401, с неверной подписью → 401,
   `GET` → 405/404.

## Вне объёма
- Обработка `message.received`/`message.sent` по существу — T-049.
- Отправка ответов — T-050. UI — T-051.

## Критерии приёмки
- [ ] Подпись проверяется от raw body через timingSafeEqual; без/с неверной подписью → 401
- [ ] Роут отвечает 200 мгновенно (обработка в waitUntil), ошибки обработчика не роняют ACK
- [ ] `account.connected` активирует запись тенанта идемпотентно;
      `account.disconnected(unintentional)` ставит `error` и уведомляет владельца в Telegram
- [ ] Неизвестные события/аккаунты не ломают обработку (200 + лог)
- [ ] Setup-скрипт идемпотентен (create либо update по имени `insta-reply`)
- [ ] Секрет вебхука не попадает в логи и вывод скрипта
- [ ] lint, typecheck зелёные; manual-test написан
