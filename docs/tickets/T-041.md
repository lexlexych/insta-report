# T-041 · Онбординг 2.1 — маршрут без IG-шагов, кнопка «Назад», полноэкранное поле базы знаний

> **Перед началом:** прочитай `docs/plan.md` §1, §5 и `docs/tickets/OVERRIDES.md`.
> Зависимости: T-038 (визард онбординга 2.0), T-039 (демонтаж LLM-генерации).
> Тикет упрощает маршрут визарда T-038: экраны 3а «Заявка подана» (`ig_wait`) и 3б «Подключение
> Instagram» (`ig_connect`) демонтируются. Подключение Instagram переезжает из онбординга на
> финальный экран и дашборд (T-042, T-043, T-044). Финальный экран перерабатывается отдельным
> тикетом T-042 — здесь его содержимое не трогать.

## Цель

Маршрут онбординга: `sphere → business → knowledge → finish → done` (без IG-шагов); кнопка
«Назад» слева от «Далее» на экранах, где есть куда возвращаться; поле базы знаний растянуто на
всю доступную высоту экрана, страница не прокручивается.

## Маршрут

1. Убрать из визарда экраны 3а (`ig_wait`) и 3б (`ig_connect`) вместе с их состояниями и
   ветвлением. Экран 2 (business) по-прежнему выполняет `PATCH tenant { orgName }` +
   `POST /api/miniapp/onboarding/ig-account`, но любой успешный ответ (`pending` ИЛИ `approved`)
   ведёт на шаг `knowledge`. Ответ 409 `{ code: 'taken' }` → инлайн-ошибка под полем, остаёмся
   на экране 2 (как раньше). Бэкенд-логику заявки НЕ менять: pending-заявка уведомляет админов
   (T-037), предодобренный ник привязывается к тенанту (`bindTenant`).
2. Допустимые переходы `onboarding_step`: `sphere → business → knowledge → finish → done`.
   В Zod-enum PATCH-роута `/api/miniapp/tenant` значения `ig_wait` / `ig_connect` исключить —
   новые записи этих шагов невозможны.
3. Legacy-состояния: тенанты, оставшиеся на `ig_wait` / `ig_connect` после T-038, при загрузке
   онбординга трактуются как `knowledge` (маппинг при чтении в гарде/визарде; при первом же
   переходе шаг перезапишется валидным значением). Если в БД на колонку наложен check/enum со
   старыми значениями — не трогать (старые значения остаются допустимыми для чтения); миграция
   данных не требуется.

## Кнопка «Назад»

4. В фиксированной нижней панели слева от «Далее» — кнопка «Назад» (вторичный стиль, i18n):
   - `business` → `sphere`; `knowledge` → `business`. На экранах `sphere` и `finish` кнопки
     «Назад» нет.
   - «Назад» всегда активна; выполняет `PATCH tenant { onboardingStep: <предыдущий> }` и
     показывает предыдущий экран. Введённые данные при возврате не теряются: в рамках сессии —
     локальное состояние страницы; после перезапуска мини-аппа — данные из API (как в T-038).
   - Повторное прохождение экрана 2 с тем же ником не создаёт дубликат заявки (идемпотентность
     роута `ig-account` сохраняется).

## Экран «Опишите ваш бизнес» (`knowledge`)

5. Раскладка без прокрутки страницы: контейнер экрана занимает ровно высоту вьюпорта
   (flex-колонка: заголовок + подсказка → textarea `flex-1` → фиксированная панель кнопок).
   Прокручивается ТОЛЬКО текст внутри textarea (`overflow-y: auto`); авто-рост textarea из
   T-038 убрать. Использовать динамическую высоту вьюпорта (`100dvh` либо viewport из
   Telegram SDK), чтобы при открытой клавиатуре панель кнопок оставалась доступной.
6. i18n ru/de для всех новых строк; валидации и логика disabled-состояния «Далее»/«Завершить»
   на остальных экранах — без изменений (T-038).
7. Manual-test `docs/manual-tests/T-041.md`: сквозной проход по новому маршруту в обоих
   вариантах (ник не в реестре / предодобренный — оба ведут на `knowledge` без промежуточных
   экранов; уведомление админам при pending уходит); «Назад» с экранов 2 и 4 и сохранение
   введённых данных; на `knowledge` страница не прокручивается, длинный текст скроллится внутри
   поля (проверить с открытой клавиатурой); тенант с legacy-шагом `ig_wait`/`ig_connect`
   продолжает с `knowledge`.

## Критерии приёмки

- [ ] Экраны 3а/3б удалены; после экрана 2 всегда открывается `knowledge` (при 409 `taken` —
      инлайн-ошибка, остаёмся на экране 2)
- [ ] Заявка по-прежнему создаётся/привязывается на шаге 2; при pending админам уходит
      уведомление (T-037)
- [ ] «Назад» есть на `business` и `knowledge`, отсутствует на `sphere` и `finish`; переходы и
      сохранение данных корректны
- [ ] На `knowledge` страница не прокручивается; textarea занимает всё доступное пространство,
      скроллится только текст внутри
- [ ] Тенант с legacy-шагом `ig_wait`/`ig_connect` продолжает онбординг с `knowledge`
- [ ] `pnpm lint && pnpm typecheck` зелёные; manual-test написан
