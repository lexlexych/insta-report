# T-008 · LLM-клиент

> **Перед началом:** прочитай `docs/plan.md` §2, §5. Зависимости: T-001 (+ T-004 для usage). Единственная точка вызова LLM во всём проекте.

## Цель
`src/lib/llm/client.ts`: обёртка над OpenAI-совместимым API (OpenRouter) c ретраями, JSON-режимом и учётом токенов.

## Шаги выполнения
1. Установи `openai`. Ленивый синглтон: `new OpenAI({ baseURL: env.LLM_BASE_URL, apiKey: env.LLM_API_KEY })`.
2. `type LlmOpts = { model: string; system: string; user: string; temperature?: number; maxTokens?: number; tenantId?: string }`.
3. `async function complete(opts: LlmOpts): Promise<{ text: string; usage: { inTok: number; outTok: number } }>`:
   - `chat.completions.create` c messages [{role:'system'},{role:'user'}], timeout 30_000;
   - ретраи: на 429 и 5xx — до 2 повторов c backoff 1s/3s (+джиттер); на 4xx — не ретраить, бросать `LlmError`;
   - пустой `choices[0].message.content` → LlmError('empty');
   - если передан tenantId — `await usageStats.increment(tenantId, { llm_calls: 1, tokens_in, tokens_out })` (ошибку учёта логировать, но не пробрасывать).
4. `async function completeJSON<T>(opts: LlmOpts, schema: ZodSchema<T>): Promise<{ data: T; usage }>`:
   - добавляет `response_format: { type: 'json_object' }` и к system — строку «Отвечай строго одним JSON-объектом без пояснений и markdown»;
   - парсинг: срежь возможные ```json-заборы, `JSON.parse`, `schema.parse`;
   - при ошибке парсинга — ОДИН повторный вызов c добавлением в user: «Предыдущий ответ не был валидным JSON по схеме. Верни только JSON.»; вторая неудача → `LlmError('bad_json')`.
5. Экспортируй константы моделей: `MODEL_CLASSIFY = env.LLM_MODEL_CLASSIFY`, `MODEL_DRAFT = env.LLM_MODEL_DRAFT`.
6. Тесты `tests/llm.client.test.ts` (мок `openai` через vi.mock):
   - 429 затем 200 → успех, два вызова;
   - 400 → LlmError без ретрая;
   - completeJSON: первый ответ «мусор», второй валидный → data распарсена, вызовов два;
   - оба невалидны → LlmError('bad_json');
   - usage прокинут в мок usageStats.increment.
