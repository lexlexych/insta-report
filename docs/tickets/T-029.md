# T-029 · Структурированное логирование и алерты

> **Перед началом:** прочитай `docs/plan.md` §2.3, §5. Зависимости: T-021.

## Цель
Единый логгер c контекстом и редактированием секретов; алерты админам о критических ошибках пайплайна.

## Шаги выполнения
1. Установи `pino`. `src/lib/log.ts`:
   - базовый логгер: JSON, level из `LOG_LEVEL` (добавь в env.ts, default 'info');
   - `redact`: пути `token`, `accessToken`, `appSecret`, `authorization`, `*.access_token`, `initData`, `session` → '[REDACTED]';
   - `logger.child({ tenantId, conversationKey, requestId })` — хелпер `withCtx(ctx)`;
   - requestId: генерируй uuid в apiHandler (T-007) и в handleIgEvent, прокидывай в child.
2. Замени все console.* в `src/` на логгер (grep-проверка). В `logPipelineError` (T-011) — `logger.error` + алерт.
3. Алерты: `src/lib/alerts.ts` — `alertAdmins(kind: string, message: string)`:
   - отправка каждому id из `env.ADMIN_TELEGRAM_IDS` (split, trim, фильтр пустых) через tg api; ошибки отправки только логировать;
   - троттлинг 1 сообщение в 60 c на kind: таблица не нужна — используй `rate_limits` из T-030, а до его готовности — in-memory Map c timestamp (пометь TODO: после T-030 перевести на общий лимитер; если T-030 уже выполнен — сразу используй его);
   - подключи alertAdmins в: logPipelineError (kind='pipeline'), cron-провалы (T-015, kind='token_refresh'), необработанные исключения apiHandler (kind='api_500').
4. Тесты `tests/log.test.ts`:
   - объект c полем accessToken → в сериализованном логе '[REDACTED]' и нет исходной строки;
   - alertAdmins: два вызова одного kind подряд → одна отправка; разные kind → две;
   - ADMIN_TELEGRAM_IDS='' → ноль отправок, ноль исключений.
5. Добавь ESLint правило `no-console: error` для `src/**` (allow в scripts/).

## Критерии приёмки
- [ ] `grep -rn "console\." src/` пуст; ESLint страхует
- [ ] Секреты редактируются в любом месте объекта лога (тест)
- [ ] Критические ошибки доходят до админов не чаще 1/мин на тип
