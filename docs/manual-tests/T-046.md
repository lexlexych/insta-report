# Ручной тест T-046 · Zernio: миграция БД и репозиторий

## Что сделано

Добавлен реестр подключения Instagram через Zernio: он связывает тенанта с профилем Zernio и
подключённым аккаунтом. Черновики теперь отмечают провайдера (`meta` или `zernio`) и при необходимости
хранят идентификатор беседы Zernio.

## Подготовка

- ветка: `git fetch && git checkout ticket/T-046 && pnpm install`
- env: заполните `SUPABASE_URL` и `SUPABASE_SECRET_KEY` в локальном `.env` значениями dev-проекта из
  Supabase Dashboard: **Project Settings → API**. Для этого теста не нужны ключи Zernio, Meta или Telegram.
- сервисы: используйте отдельный dev-проект Supabase. Выполните `pnpm db:push`; веб-сервер запускать не нужно.

## Проверки

1. Выполните `pnpm db:push` на чистом dev-проекте, где применены предыдущие миграции → миграция
   `0013_zernio.sql` применяется без ошибок; `ig_connections` и `ig_accounts` не изменяются.
2. В Supabase Dashboard откройте **Table Editor** → `zernio_accounts` → существуют поля `id`, `tenant_id`,
   `platform`, `zernio_profile_id`, `zernio_account_id`, `username`, `status`, `connected_at`,
   `disconnect_reason`, `created_at`; `platform` по умолчанию равен `instagram`, а `status` — `pending`.
3. В SQL Editor выполните `select indexname from pg_indexes where tablename = 'zernio_accounts' order by indexname;`
   → среди результатов есть `zernio_accounts_tenant_platform_uniq`,
   `zernio_accounts_account_id_uniq` и `zernio_accounts_profile_idx`.
4. В SQL Editor выполните `select column_name, column_default from information_schema.columns where table_name = 'drafts' and column_name in ('provider', 'zernio_conversation_id') order by column_name;`
   → возвращаются обе колонки; у `provider` указан default `meta`, а `zernio_conversation_id` допускает `NULL`.
5. В SQL Editor выполните `select relrowsecurity from pg_class where oid = 'zernio_accounts'::regclass;` →
   возвращается `true`; в Dashboard у таблицы нет RLS policies.
6. Выполните `pnpm tsx scripts/manual/T-046-demo.ts` → выводится строка отключённого аккаунта со
   `status: "disconnected"`, затем `T-046 demo OK`. Скрипт создаёт временного тенанта, проверяет
   pending → active → повторный active → поиск по account ID → disconnected и в конце удаляет все данные.
7. В SQL Editor создайте временного тенанта и строку `zernio_accounts` для него, затем удалите тенанта
   командой `delete from tenants where id = '<tenant-id>';` → связанная строка `zernio_accounts` удаляется
   автоматически по внешнему ключу.

## Негативные проверки

1. В SQL Editor создайте две строки `zernio_accounts` для разных тенантов, затем попробуйте присвоить обеим
   одинаковый непустой `zernio_account_id` → PostgreSQL сообщает ошибку уникальности, вторая запись не меняется.
   При вызове `zernioAccounts.activate` такая ошибка преобразуется в `ZernioAccountConflictError`.

## Как понять, что тикет НЕ работает

Если `pnpm db:push` завершается ошибкой, проверьте, что миграции до `0012_drop_org_description.sql` применены
в dev-проекте. Если demo-скрипт не проходит, проверьте service-role key и строки временного тенанта в Table Editor.
Отсутствие любого из трёх индексов, двух полей `drafts`, включённого RLS или каскадного удаления означает, что
миграция применена не полностью.
