# Ручной тест T-023 · UI управления категориями

## Что сделано
- Раздел «Категории» в Mini App перестал быть заглушкой: на `/app/labels` теперь полноценный CRUD меток (имя, описание, инструкция, порядок).
- Появились API `GET/POST /api/miniapp/labels`, `PUT/DELETE /api/miniapp/labels/[id]`, `PUT /api/miniapp/labels/reorder` — с проверкой владения тенантом, Zod-валидацией и защитой служебной метки «Без категории».
- «Без категории» неудаляема и непереименуема на всех уровнях (UI/API/репозиторий) и всегда стоит последней; порядок меток задаётся кнопками ↑/↓ и сохраняется. Автотестов нет (OVERRIDES) — краевые случаи проверяются шагами ниже.

## Подготовка
- ветка: `git fetch && git checkout claude/categories-section-status-02a1r8 && pnpm install`
- env: как для остальных miniapp-проверок (см. T-011/T-016/T-021) — заполненные в `.env.example` переменные Supabase (URL + ключи), Telegram-бота (токен из BotFather) и секрет initData Mini App. Отдельных новых переменных T-023 не вводит.
- сервисы:
  - dev-проект Supabase с применёнными миграциями: `pnpm db:push` (нужны таблицы `labels`, `drafts`; сид создаёт «Без категории» с `sort=999`).
  - `pnpm dev`; открыть Mini App через Telegram-бота (чтобы пройти аутентификацию тенанта). Для быстрых негативных проверок API удобно использовать консоль DevTools уже открытого (авторизованного) Mini App — запросы `fetch` унаследуют сессию.

## Проверки (Действие → Ожидаемо)
1. **Заглушки больше нет** [регресс исходной проблемы]: откройте нижнюю вкладку «Категории» → ожидаемо вместо текста «Раздел «Категории» в разработке.» показан список категорий; внизу карточка **«Без категории»** с иконкой замка 🔒 и без кнопок редактирования/удаления/перемещения.
2. **Создание категории** [AC: базовый CRUD]: нажмите FAB «+ Категория» → открывается нижняя форма (bottom sheet) с тремя полями и подсказками-примерами: name — «Запись на услугу», description — «Клиент хочет записаться…», instruction — «Уточни желаемую дату и услугу…». Заполните имя (напр. «Запись»), сохраните → ожидаемо форма закрылась, новая карточка появилась в списке **выше** «Без категории», с описанием серым текстом.
3. **Порядок и его влияние на GET** [AC2]: создайте 2–3 категории. Кнопками ↑/↓ поменяйте их порядок → ожидаемо порядок в списке меняется сразу. Перезагрузите Mini App (или вкладку) → ожидаемо новый порядок сохранился. В DevTools → Network ответ `GET /api/miniapp/labels` возвращает метки в этом же порядке (`sort` по возрастанию), «Без категории» — последней.
4. **Редактирование** [AC: CRUD]: у обычной категории нажмите ✏️, измените имя/описание/инструкцию, сохраните → ожидаемо изменения отражены в карточке и сохраняются после перезагрузки.
5. **Удаление с подтверждением** [AC: CRUD + on delete set null]: у обычной категории нажмите 🗑 → появляется запрос подтверждения; подтвердите → ожидаемо карточка исчезла. Если к этой метке были привязаны черновики (`drafts.label_id`), в БД у них `label_id` стал `NULL` (метка удалена, черновики не удалены) — за счёт FK `on delete set null`.
6. **Замок «Без категории» в UI** [AC1 — уровень UI]: у карточки «Без категории» нет и не появляется кнопок ✏️/🗑/↑/↓ ни при каких действиях.

## Негативные проверки
1. **Дубль имени → 409** [краевой случай из «Тесты»]: попробуйте создать (или переименовать) категорию с именем, которое уже есть → ожидаемо сохранение не проходит, показывается ошибка «имя занято» (ответ API — HTTP 409), список не меняется.
2. **Переименование «Без категории» через API → 403** [AC1 — уровень API/repo]: в DevTools-консоли авторизованного Mini App выполните
   `fetch('/api/miniapp/labels/<ID_БЕЗ_КАТЕГОРИИ>', {method:'PUT', headers:{'Content-Type':'application/json'}, body:JSON.stringify({name:'Хочу переименовать'})}).then(r=>r.status)`
   (ID возьмите из ответа `GET /api/miniapp/labels`) → ожидаемо статус **403** (`forbidden_default_label`), имя метки не изменилось.
3. **Удаление «Без категории» через API → 403** [AC1]: тем же способом `fetch('/api/miniapp/labels/<ID_БЕЗ_КАТЕГОРИИ>', {method:'DELETE'}).then(r=>r.status)` → ожидаемо **403**, метка на месте.
4. **Чужая метка → 404** [безопасность/cross-tenant]: подставьте случайный/чужой UUID в `PUT` или `DELETE /api/miniapp/labels/<uuid>` → ожидаемо **404** (метку нельзя менять из другого тенанта), а не 200/500.
5. **Валидация полей** [краевой случай из «Тесты»]: попробуйте создать категорию с пустым именем или именем длиннее 50 символов (либо description > 300 / instruction > 1000) → ожидаемо сохранение отклонено (ответ 400), понятная ошибка формы.

## Как понять, что тикет НЕ работает
- Вкладка «Категории» всё ещё показывает «в разработке»: убедитесь, что вы на ветке `claude/categories-section-status-02a1r8`, `pnpm dev` перезапущен, а `app/(miniapp)/app/labels/page.tsx` больше не рендерит `PlaceholderPage`.
- Порядок «слетает» после перезагрузки: проверьте, что кнопки ↑/↓ шлют `PUT /api/miniapp/labels/reorder` (вкладка Network) и что `GET` сортирует по `sort`; в таблице `labels` поле `sort` должно меняться.
- «Без категории» удаляется/переименовывается или не последняя: проверьте в БД её `name` (`Без категории`, точное совпадение) и `sort=999`; ответы API на PUT/DELETE по её id должны быть 403.
- Ошибки 500 вместо 409/403/404: смотрите логи `pnpm dev` и Network — коды должны быть осмысленными (409 дубль имени, 403 дефолтная метка, 404 чужой/несуществующий id, 400 валидация).
