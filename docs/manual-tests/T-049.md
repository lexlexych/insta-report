# Ручной тест T-049 · Zernio: входящие сообщения в общий pipeline

## Что сделано

Входящие DM и исходящие echo-события Zernio приводятся к формату существующего Instagram
pipeline. Для входящих переиспользуются получение контекста, классификация, генерация и доставка
карточки в Telegram; новый код не дублирует эти этапы. Черновик сохраняет провайдера `zernio` и
ID его переписки, чтобы следующий тикет мог отправить ответ через Zernio.

## Подготовка

- ветка: `git fetch && git checkout zernio && pnpm install`
- env: заполните обязательные переменные из `.env.example`, а также `ZERNIO_API_KEY` и
  `ZERNIO_WEBHOOK_SECRET`. Значения секретов не вставляйте в терминал, документацию или скриншоты.
- сервисы: примените миграции T-046 командой `pnpm db:push`; запустите `pnpm dev`; поднимите
  публичный HTTPS-туннель на порт 3000 и укажите его адрес в `APP_BASE_URL`. Выполните
  `pnpm tsx scripts/manual/zernio-setup-webhook.ts`, чтобы глобальный webhook Zernio был активен.
- подготовьте тестовый Instagram-аккаунт, подключённый через Zernio по T-047/T-048, и второй
  Instagram-аккаунт для отправки DM. Для проверки общего дедупа дополнительно подключите тот же
  бизнес-аккаунт прямым Meta-способом; это необязательно для остальных шагов.

## Проверки

1. Выполните `pnpm tsx scripts/manual/T-049-demo.ts` → все строки начинаются с `OK`, в том числе
   проверка входящего сообщения, вложения, echo, повтора `mid`, чужого аккаунта, Zernio-контекста
   и полей черновика. Скрипт использует моки и не обращается к сети, БД или настоящему Telegram.
2. С постороннего Instagram-аккаунта отправьте тестовому бизнес-аккаунту обычный текстовый DM →
   в Telegram владельца приходит одна карточка с текстом клиента, категорией и сгенерированным
   черновиком; в её переписке отображается контекст. В журнале Zernio есть `message.received`.
3. Отправьте клиенту только изображение без подписи → приходит карточка, где входящее сообщение
   показано как `[вложение]`, а черновик просит клиента пояснить содержание вложения.
4. Повторно доставьте тот же `message.received` из журнала вебхука Zernio (или инициируйте retry
   тем же событием) → второй Telegram-карточки нет. В `processed_events` существует одна запись
   с соответствующим Meta `platformMessageId`.
5. Пока карточка из шага 2 ожидает ответа, ответьте этому клиенту вручную из приложения Instagram
   владельца → после `message.sent` pending-карточка исчезает, а в `message_log` появляется
   запись `manual`. Если Zernio не присылает `message.sent` для ручного ответа в пилоте, запишите
   этот факт в заметках эпика: это известный открытый вопрос research §9.2.
6. При одновременно активном прямом Meta-подключении того же аккаунта отправьте один новый DM →
   независимо от порядка двух вебхуков появляется ровно одна карточка. Это подтверждает общий
   дедуп по Meta `platformMessageId`.
7. Отключите Zernio-аккаунт, затем отправьте ему DM → новая карточка не создаётся; в серверном
   логе есть `[zernio/message] skip inactive or unknown account`.

## Негативные проверки

1. В Zernio Dashboard подайте событие не для Instagram или сообщение без `platformMessageId` →
   endpoint отвечает `200`, но карточка не создаётся; лог содержит `[zernio/message] ignored`.
2. Отправьте чистый репост поста или Reels без текста → после записи `mid` в дедуплицаторе
   карточка не создаётся; это совпадает с поведением прямого Meta-webhook.
3. Повторите HMAC-проверку из manual-test T-048 с неверной либо отсутствующей подписью → endpoint
   возвращает `401`, а pipeline не запускается.

## Как понять, что тикет НЕ работает

Если после DM нет карточки, проверьте активность строки `zernio_accounts`, доставку `message.received`
в Dashboard и серверный лог с префиксами `[wh/zernio]`, `[zernio/message]` и `[pipeline]`. Если
дубли появляются, сравните `platformMessageId` обеих доставок и уникальную запись в
`processed_events`. Если в карточке нет истории, проверьте доступность Inbox addon и ошибки
Zernio API (таймаут/429): они логируются pipeline и не должны содержать ключей API.
