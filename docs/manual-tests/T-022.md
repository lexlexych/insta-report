# Ручной тест T-022 · Echo владельца отменяет черновик

## Что сделано
- Echo-событие Instagram от бизнес-аккаунта теперь ищет pending-черновик по conversation key, переводит его в `cancelled`, удаляет Telegram-карточку и пишет `message_log` с направлением `manual`.
- После отправки черновика через кнопку Telegram возвращённые Meta `mid` сразу регистрируются в `processed_events`, поэтому собственный echo отправки отсекается дедупликацией и не отменяет следующий свежий черновик.
- Автотесты по OVERRIDES не добавлялись; ключевые edge cases вынесены в demo-скрипт ниже.

## Подготовка
- ветка: `git fetch && git checkout ticket/T-022 && pnpm install`
- env для локального demo-скрипта не нужен: скрипт сам выставляет dummy-переменные и не ходит во внешние сервисы.
- для полного ручного прогона нужен уже настроенный dev-проект Supabase, Telegram-бот, Mini App, Meta/Instagram webhook и переменные из `.env.example`, как в manual-tests T-011, T-016, T-020 и T-021.
- сервисы для полного прогона: `pnpm dev`; для вебхуков — публичный туннель на порт 3000; для БД — применённые миграции через `pnpm db:push` на dev-проекте Supabase.

## Проверки
1. **Локальная контрактная проверка:** выполните `pnpm tsx scripts/manual/T-022-demo.ts` → ожидаемо выводит `pending echo`, `no pending echo`, `own echo dedup` и финальную строку `T-022 demo OK`.
2. **Ручной echo отменяет карточку:** отправьте входящее Instagram-сообщение от клиента так, чтобы бот создал pending-карточку в Telegram; затем ответьте этому клиенту вручную из Instagram Business/Meta Inbox → ожидаемо Telegram-карточка удалена, в таблице `drafts` у этого conversation key статус `cancelled`, в `message_log` появилась запись `direction=manual` с текстом ручного ответа или `[вложение]` для ответа без текста.
3. **Нет pending — no-op:** повторно отправьте echo/ручной ответ в диалоге, где уже нет pending-карточки → ожидаемо приложение не падает, Telegram delete не повторяется для старой карточки, новых черновиков/логов `manual` для отсутствующего pending нет.
4. **Удалённая Telegram-карточка не ломает поток:** создайте pending-карточку, удалите сообщение карточки в Telegram вручную, затем ответьте клиенту из Instagram → ожидаемо обработчик не падает на ошибке Telegram `message to delete not found`, а черновик всё равно становится `cancelled`.
5. **Собственный echo после отправки не отменяет новый черновик:** создайте pending-карточку, нажмите `Отправить`; дождитесь статуса `sent`, затем отправьте новое входящее сообщение от клиента, чтобы появилась новая pending-карточка → ожидаемо echo сообщения, отправленного API, не удаляет новую карточку; в `processed_events` есть `mid`, возвращённый Meta при отправке.

## Расширенный ручной прогон полного пайплайна из T-031
1. **Happy path:** входящий текст клиента → Telegram получает карточку с черновиком LLM; нажмите `Отправить` при отсутствии ручного ответа в Instagram → ожидаемо IG `/messages` отправлен, статус черновика `sent`, карточка отредактирована на месте в шапку «✅ отправлено в HH:MM @получатель» (без кнопок), в `message_log` есть `in` и `out`.
2. **Echo отменяет:** входящий текст клиента → есть pending-карточка; ручной ответ владельца в Instagram → ожидаемо Telegram delete вызван наблюдаемо удалением карточки, статус `cancelled`, IG send по кнопке не выполнялся.
3. **Дубликат вебхука:** отправьте/переиграйте один и тот же webhook с одинаковым `mid` два раза → ожидаемо создана ровно одна Telegram-карточка и один pending-черновик.
4. **Ручной ответ между карточкой и кнопкой:** создайте pending-карточку, вручную ответьте клиенту в Instagram, затем нажмите старую кнопку `Отправить` → ожидаемо статус `skipped_manual`, IG send не вызывается, карточка показывает отмену из-за ручного ответа.
5. **Второе сообщение клиента замещает черновик:** отправьте входящее сообщение №1 → появилась карточка №1; отправьте входящее сообщение №2 с другим `mid` в том же диалоге → ожидаемо карточка №1 удалена, остался один новый pending-черновик и новая карточка.

## Негативные проверки
1. Отправьте webhook без `message` или с уже обработанным `mid` → ожидаемо обработчик мягко выходит, `handleEcho` не вызывается, текущие pending-карточки не меняются.
2. Вручную ответьте вложением без текста → ожидаемо pending-карточка отменяется, а в `message_log.text` записано `[вложение]`.

## Как понять, что тикет НЕ работает
- Telegram-карточка остаётся после ручного ответа владельца: проверьте логи `pnpm dev`, запись в `drafts.status` и соответствие `conversation_key` формату `<business_ig_id>:<client_igsid>`.
- Новая карточка исчезает сразу после отправки старого черновика через кнопку: проверьте, что `sendMessage` вернул непустой `mid`, а `processed_events` содержит этот `mid` до прихода echo-webhook.
- Обработчик падает при удалённой Telegram-карточке: проверьте текст ошибки Telegram; `message to delete not found` должен подавляться в `deleteMessageSafe`.
