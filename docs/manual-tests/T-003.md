# Ручной тест T-003 · Модуль шифрования секретов

## Что сделано

Добавлен `src/lib/crypto.ts` — AES-256-GCM шифрование строк с версионированным
форматом `v1:<iv>:<tag>:<ciphertext>` (все части в base64), ключ берётся из
`env.ENCRYPTION_KEY`. Функции `encrypt`, `decrypt` (бросает `DecryptError` при
любой ошибке формата/тега, без утечки payload в текст ошибки), `isEncrypted`.
Добавлен скрипт генерации ключа `scripts/gen-key.ts` (команда `pnpm gen:key`)
и demo-скрипт `scripts/manual/T-003-demo.ts` для ручной проверки (модуль без UI).

## Подготовка

- ветка: `git fetch && git checkout claude/ticket-t003-pvk263 && pnpm install`
- реального `.env.local` для этой проверки не нужно — demo-скрипт сам
  подставляет фиктивные значения всех обязательных переменных окружения
  (включая случайный `ENCRYPTION_KEY`) перед импортом модуля, чтобы не падать
  на отсутствии реального `.env`.

## Проверки (каждая: Действие → Ожидаемо)

1. **Генерация ключа.**
   Действие: `pnpm gen:key`.
   Ожидаемо: в консоль печатается случайная base64-строка (декодируется в 32
   байта) и подсказка добавить её в `ENCRYPTION_KEY` в `.env.local`. Запустите
   команду ещё раз — второе значение отличается от первого (значит ключ
   действительно случайный, а не захардкожен).

2. **Демонстрация шифрования/расшифровки.**
   Действие: `pnpm tsx scripts/manual/T-003-demo.ts`.
   Ожидаемо: в выводе по порядку блоки `=== 1. ... ===` … `=== 6. ... ===`, в
   каждом — строки `OK: ...` (без единого `FAIL:`), а в самом конце — `Итог`
   и строка `Все проверки пройдены`. Код возврата процесса — `0`
   (`echo $?` после запуска на Linux/macOS).

3. **Roundtrip для разных видов строк (блок 1 вывода).**
   Ожидаемо: для `ascii`, `кириллица`, `emoji` и `пустая строка` — расшифрованное
   значение совпадает с исходным (`decrypted` в выводе равно `plain`).

4. **Разные IV у одинакового plaintext (блок 2 вывода).**
   Ожидаемо: `encrypt(#1)` и `encrypt(#2)` для одного и того же текста —
   заметно разные строки (разные `iv`/`tag`/`ciphertext` части), проверка
   помечена `OK`.

5. **typecheck/lint.**
   Действие: `pnpm lint && pnpm typecheck`.
   Ожидаемо: обе команды завершаются без ошибок.

## Негативные проверки

1. **Порча `tag`-части (блок 3 вывода).**
   Ожидаемо: в выводе показаны «оригинал» и «испорчен» (отличается один
   символ в tag-сегменте), затем `OK: DecryptError выброшен как ожидалось
   (испорчен tag)` — никакого «тихого» неверного результата, только ошибка.

2. **Порча `ciphertext`-части (блок 4 вывода).**
   Ожидаемо: аналогично предыдущему — `OK: DecryptError выброшен как ожидалось
   (испорчен ciphertext)`.

3. **`decrypt('мусор')` — произвольная строка не в нашем формате (блок 5 вывода).**
   Ожидаемо: `OK: DecryptError выброшен как ожидалось (мусор на входе)`.

4. **Секрет не утекает в текст ошибки.**
   Действие: посмотрите на сообщения об ошибках в блоках 3-5 вывода (Node сам
   печатает `error instanceof DecryptError`, а не текст исключения) — при
   желании откройте `src/lib/crypto.ts` и убедитесь, что `DecryptError`
   (строки ~20-27) не принимает и не сохраняет payload в `message`.
   Ожидаемо: сообщение `DecryptError` — фиксированный текст
   («Failed to decrypt payload: ...»), никогда не содержит исходную
   зашифрованную строку или её фрагменты.

## Как понять, что тикет НЕ работает

- `pnpm tsx scripts/manual/T-003-demo.ts` завершается с ненулевым кодом и/или
  в выводе есть хотя бы одна строка `FAIL:` — читайте секцию «Провалено
  проверок» в конце вывода, там перечислены конкретные упавшие проверки.
- `pnpm gen:key` печатает одно и то же значение при повторных запусках →
  ключ не случайный, критическая проблема безопасности.
- Блок 3 или 4 не бросает `DecryptError` (`FAIL: decrypt() с испорченным ...
  должен был бросить DecryptError, но не бросил`) → аутентификация GCM не
  проверяется, значит подделанные данные расшифровываются молча — критическая
  проблема безопасности, тикет не принимать.
- `pnpm typecheck` падает на `src/lib/crypto.ts` или `scripts/gen-key.ts` →
  смотреть текст ошибки TypeScript в этих файлах.
