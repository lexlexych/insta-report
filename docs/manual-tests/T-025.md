# Ручной тест T-025 · Бэкенд тест-чата (симулятор)

## Что сделано

Добавлен серверный endpoint `POST /api/miniapp/simulator/message`, который принимает виртуальную историю тест-чата из Mini App и прогоняет её через общий pipeline: `deriveContext` → `labels.listByTenant` → `classify` → `generateDraft`. Endpoint не хранит историю на сервере, проверяет наличие базы знаний и считает отдельный суточный лимит симулятора в `usage_stats.simulator_calls`.

## Подготовка

- Ветка: `git fetch && git checkout ticket/T-025 && pnpm install`.
- Env: заполнить обычные переменные приложения из `.env.example` для Supabase, Mini App JWT и LLM. Для проверки лимита временно установить `SIMULATOR_DAILY_LIMIT=3`.
- БД: применить миграции в dev-проект Supabase (`pnpm db:push` или эквивалентный push из Supabase CLI). После применения в таблице `usage_stats` должна появиться колонка `simulator_calls int not null default 0`, а RPC `increment_usage` должен принимать параметр `p_simulator_calls`.
- Данные: нужен tenant с валидной Mini App session-cookie `session`, хотя бы одной меткой и заполненной базой знаний (`knowledge_base`). Проще всего пройти существующий онбординг Mini App и затем использовать cookie из браузера DevTools.
- Сервис: запустить `pnpm dev`.

## Проверки

1. **Успешный прогон pipeline.**
   - Действие: выполнить запрос, подставив cookie реального tenant:
     ```bash
     curl -i http://localhost:3000/api/miniapp/simulator/message \
       -H 'Content-Type: application/json' \
       -H 'Cookie: session=<SESSION_COOKIE>' \
       --data '{"messages":[{"role":"client","text":"Здравствуйте, сколько стоит консультация?"}]}'
     ```
   - Ожидаемо: HTTP 200; JSON содержит `label.id`, `label.name`, непустой `draft`, `usage.simulatorCalls: 1`, `usage.dailyLimit: 3`. В Supabase в `usage_stats` за сегодняшний день у tenant значение `simulator_calls` увеличилось на 1. Это покрывает критерий «ноль дублирования пайплайн-логики»: ответ появился через тот же classify+draft pipeline, а не через отдельную заглушку.
2. **Корректная деривация истории и pendingText.**
   - Действие: отправить историю, где последнее сообщение бизнеса отделяет старый вопрос клиента от нового:
     ```bash
     curl -i http://localhost:3000/api/miniapp/simulator/message \
       -H 'Content-Type: application/json' \
       -H 'Cookie: session=<SESSION_COOKIE>' \
       --data '{"messages":[{"role":"client","text":"Старый вопрос про адрес"},{"role":"assistant","text":"Мы на главной улице"},{"role":"client","text":"А завтра вы открыты?"},{"role":"client","text":"И во сколько?"}]}'
     ```
   - Ожидаемо: HTTP 200; черновик отвечает именно на последние подряд client-сообщения про завтрашний график, а не на старый вопрос про адрес. В логах приложения нет ошибок `Unhandled API error`.
3. **Суточный лимит.**
   - Действие: при `SIMULATOR_DAILY_LIMIT=3` выполнить всего три успешных запроса за текущий день, затем четвёртый такой же запрос.
   - Ожидаемо: первые три запроса возвращают HTTP 200; четвёртый возвращает HTTP 429 и JSON `{"code":"daily_limit"}`. Значение `usage_stats.simulator_calls` не увеличивается после отклонённого четвёртого запроса. Это покрывает критерий «дневной лимит работает» с лимитом, сниженным через env по OVERRIDES.

## Негативные проверки

1. **Tenant без базы знаний.**
   - Действие: для tenant очистить `knowledge_base` через существующий endpoint/SQL в dev-БД и повторить запрос к simulator.
   - Ожидаемо: HTTP 409 и JSON `{"code":"onboarding_required"}`; LLM не вызывается, `usage_stats.simulator_calls` не увеличивается.
2. **Невалидное тело запроса.**
   - Действие: отправить пустой массив или слишком длинный текст:
     ```bash
     curl -i http://localhost:3000/api/miniapp/simulator/message \
       -H 'Content-Type: application/json' \
       -H 'Cookie: session=<SESSION_COOKIE>' \
       --data '{"messages":[]}'
     ```
   - Ожидаемо: HTTP 400 и JSON `{"ok":false,"error":"malformed"}`.

## Как понять, что тикет НЕ работает

- HTTP 500 или `Unhandled API error` в терминале `pnpm dev` — смотреть стек ошибки, особенно env, Supabase RPC `increment_usage` и LLM-настройки.
- HTTP 200, но `usage_stats.simulator_calls` не растёт — проверить, применена ли миграция `0006_simulator_usage.sql` и обновлена ли функция `increment_usage`.
- 4-й запрос при `SIMULATOR_DAILY_LIMIT=3` не даёт 429 — проверить, что dev-сервер перезапущен после изменения env и что запросы идут под одним tenant.
- Ответ игнорирует последние client-сообщения после сообщения бизнеса — проверить преобразование `role: assistant` в сторону «Бизнес» и `deriveContext`.
