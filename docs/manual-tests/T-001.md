# Ручной тест T-001 · Скаффолд проекта

## Что сделано (по-человечески)
Создан каркас репозитория: Next.js 15 (App Router) + TypeScript strict + Tailwind + pnpm.
Настроены ESLint/Prettier, типобезопасный модуль `src/lib/env.ts` (Zod-валидация всех переменных
окружения с ленивой инициализацией), CI на GitHub Actions (lint + typecheck), структура каталогов
из plan.md §5 и заглушка страницы Mini App `/app`. Автотесты в проект НЕ добавлялись (по OVERRIDES).

## Подготовка
- ветка: `git fetch && git checkout claude/status-command-5tw1if && pnpm install`
  (в этом проекте ветку `ticket/T-001` заменяет `claude/status-command-5tw1if` — ограничение облачного окружения).
- Node: версия 20+ (`node -v`). Менеджер пакетов — pnpm (`corepack enable` при необходимости).
- env: **для этой проверки заполнять `.env` НЕ нужно** — ни одна из проверок T-001 не требует реальных
  секретов. Модуль env инициализируется лениво, поэтому `pnpm dev`/`build` поднимаются без переменных.
  Файл `.env.example` в этом тикете проверяется только визуально (см. проверку 3).
- сервисы: только локальный `pnpm dev`. Внешние сервисы (Supabase, Telegram, Meta) в T-001 не участвуют.

## Проверки (каждая: Действие → Ожидаемо)

1. **AC: `pnpm dev` поднимает `/app` без ошибок.**
   Действие: `pnpm dev`, открыть в браузере `http://localhost:3000/app`.
   Ожидаемо: страница показывает текст «InstaReply — загрузка…». В терминале dev-сервера и в консоли
   браузера (DevTools → Console) нет ошибок и нет красных предупреждений о сборке.

2. **AC: `pnpm lint && pnpm typecheck` зелёные локально.**
   Действие: остановить dev (Ctrl+C), выполнить `pnpm lint && pnpm typecheck`.
   Ожидаемо: обе команды завершаются кодом 0, без ошибок и предупреждений ESLint/TypeScript.
   Дополнительно (не обязательно, для уверенности): `pnpm build` собирается успешно, маршрут `/app`
   помечается как статический.

3. **AC: в `.env.example` есть все переменные из plan.md §2.5 с комментариями.**
   Действие: открыть `.env.example`. Пересчитать переменные командой:
   `grep -E '^[A-Z_]+=' .env.example | wc -l`.
   Ожидаемо: ровно **13** переменных, у каждой есть комментарий-описание сверху:
   `TELEGRAM_BOT_TOKEN`, `TELEGRAM_WEBHOOK_SECRET`, `MINIAPP_JWT_SECRET`, `ENCRYPTION_KEY`,
   `SUPABASE_URL`, `SUPABASE_SECRET_KEY`, `LLM_BASE_URL`, `LLM_API_KEY`, `LLM_MODEL_CLASSIFY`,
   `LLM_MODEL_DRAFT`, `APP_BASE_URL`, `CRON_SECRET`, `ADMIN_TELEGRAM_IDS`.
   У секретов пример-значение `changeme`; у `ENCRYPTION_KEY` в комментарии сказано «base64 от 32 байт».

4. **AC: импорт `env` типобезопасен (автодополнение полей).**
   Действие: во временном файле-черновике (например, `app/(miniapp)/app/page.tsx`, потом откатить)
   добавить строку `const token = env.TELEGRAM_BOT_TOKEN;` с импортом `import { env } from '@/lib/env';`.
   В редакторе навести курсор на `env.` — появляется автодополнение всех 13 полей; тип `token` — `string`.
   Запустить `pnpm typecheck`.
   Ожидаемо: typecheck зелёный; поля `env.*` доступны с автодополнением и типами из Zod-схемы.
   **После проверки строку удалить** (`git checkout -- .`).

5. **CI зелёный (lint + typecheck, без test).**
   Действие: открыть вкладку **Actions** у PR этого тикета на GitHub.
   Ожидаемо: workflow `ci.yml` проходит; в нём есть шаги `pnpm lint` и `pnpm typecheck` и **нет** шага
   с `pnpm test`. Vitest в проекте отсутствует (это соответствует OVERRIDES).

## Негативные проверки (минимум одна)

1. **Тип поля `env` — строка, а не `any` (доказывает типобезопасность).**
   Действие: во временном черновике написать `const n: number = env.TELEGRAM_BOT_TOKEN;`
   (импорт `import { env } from '@/lib/env';`), выполнить `pnpm typecheck`.
   Ожидаемо: `tsc` падает с ошибкой «Type 'string' is not assignable to type 'number'» — значит поля
   действительно типизированы как `string` из Zod-схемы, а не `any`. **Строку удалить после проверки.**

2. **(Опционально) Валидация env fail-fast при неверном ключе.**
   Действие: в отдельном временном скрипте вызвать `envSchema.safeParse({...})` с
   `ENCRYPTION_KEY`, чей base64-декод даёт не 32 байта (например, `Buffer.alloc(16).toString('base64')`),
   и с отсутствующим `TELEGRAM_BOT_TOKEN`.
   Ожидаемо: оба случая дают `success: false` — refine отклоняет ключ неверной длины, а обязательная
   переменная без значения даёт ошибку валидации. (Эти краевые случаи заложены в коде `src/lib/env.ts`
   согласно OVERRIDES §1; отдельных автотестов в проекте нет.)

## Как понять, что тикет НЕ работает (типовые симптомы и куда смотреть)
- `/app` отдаёт 404 → проверь route group `app/(miniapp)/app/page.tsx` и `app/layout.tsx`.
- `pnpm dev` падает при старте с ZodError про env → регрессия ленивости: `envSchema.parse` не должен
  вызываться на верхнем уровне модуля `src/lib/env.ts` (только внутри `getEnv()` при первом обращении).
- `pnpm lint`/`typecheck` красные → смотри вывод команды: путь файла и правило/код ошибки TS.
- CI падает на `pnpm install --frozen-lockfile` → не закоммичен/устарел `pnpm-lock.yaml`.
- В `.env.example` меньше 13 переменных или нет комментариев → вернуть тикет: не закрыт AC про env.
