# Ручной тест T-005 · Каркас Telegram-бота и вебхук

## Что сделано
Добавлен центральный grammY-бот: `/start` идемпотентно создаёт/обновляет tenant по Telegram user id, сохраняет chat id и отправляет приветствие с кнопкой Mini App. Добавлен защищённый endpoint `/api/telegram`, тонкие HTML-хелперы Telegram API и скрипт `pnpm setup:telegram` для настройки webhook, команд и меню.

## Подготовка
- Ветка: `git fetch && git checkout ticket/T-005 && pnpm install`.
- Env: скопируйте `.env.example` в `.env.local` и заполните минимум:
  - `TELEGRAM_BOT_TOKEN` — токен тестового бота из BotFather.
  - `TELEGRAM_WEBHOOK_SECRET` — случайная строка, которую вы будете передавать в заголовке проверки.
  - `APP_BASE_URL` — публичный HTTPS URL приложения или туннеля, например localtunnel/cloudflared на порт 3000.
  - `SUPABASE_URL` и `SUPABASE_SERVICE_ROLE_KEY` — dev-проект Supabase с миграциями из T-002/T-004.
  - Остальные обязательные переменные из `.env.example` можно заполнить dev-значениями; `ENCRYPTION_KEY` сгенерировать через `pnpm gen:key`.
- Сервисы: примените миграции к dev Supabase (`pnpm db:push`), затем запустите приложение `pnpm dev` и публичный HTTPS-туннель на `http://localhost:3000`. Укажите URL туннеля в `APP_BASE_URL`.

## Проверки
1. Выполните `pnpm setup:telegram`.
   → Ожидаемо: команда завершается без ошибки и печатает JSON `getWebhookInfo`; поле `url` заканчивается на `/api/telegram`, а `allowed_updates` содержит `message`, `callback_query`, `my_chat_member`.
2. Выполните `pnpm setup:telegram` повторно.
   → Ожидаемо: команда снова завершается без ошибки, webhook остаётся тем же. Это подтверждает идемпотентность setup-скрипта.
3. Откройте чат с тестовым ботом в Telegram и отправьте `/start`.
   → Ожидаемо: бот отвечает приветствием на русском или немецком в зависимости от языка аккаунта Telegram; под сообщением есть кнопка Web App для открытия панели (`/app`).
4. Отправьте `/start` ещё раз тем же Telegram-аккаунтом.
   → Ожидаемо: бот отвечает повторно, но в таблице `tenants` остаётся одна строка с тем же `telegram_user_id`; поле `tg_chat_id` заполнено id текущего чата.
5. Проверьте защищённый webhook вручную: отправьте POST на `${APP_BASE_URL}/api/telegram` с заголовком `X-Telegram-Bot-Api-Secret-Token: <TELEGRAM_WEBHOOK_SECRET>` и минимальным валидным JSON update от Telegram (или используйте реальный update через Telegram после `setWebhook`).
   → Ожидаемо: запрос не получает `401`, а в логах Next.js видно, что update дошёл до grammY-обработчика.

## Негативные проверки
1. Выполните POST на `${APP_BASE_URL}/api/telegram` без заголовка `X-Telegram-Bot-Api-Secret-Token` или с неверным значением.
   → Ожидаемо: HTTP `401 Unauthorized`; update не обрабатывается, новых сообщений от бота и новых/изменённых строк в `tenants` нет.
2. Удалите сообщение бота вручную и позже используйте кодовый путь `deleteMessageSafe` для уже отсутствующего сообщения.
   → Ожидаемо: ошибка Telegram `message to delete not found` не ломает выполнение; другие ошибки Telegram должны быть видны в логах и не замалчиваться.

## Соответствие критериям приёмки
- `401` при неверном secret и прохождение update при верном secret: проверки 5 и негативная 1.
- `/start` идемпотентен, кнопка `web_app` присутствует: проверки 3 и 4.
- setup-скрипт идемпотентен: проверки 1 и 2.

## Как понять, что тикет НЕ работает
- `pnpm setup:telegram` падает с ошибкой Telegram: проверьте `TELEGRAM_BOT_TOKEN`, публичный HTTPS `APP_BASE_URL` и отсутствие лишних пробелов в `.env.local`.
- Webhook возвращает `401` для реальных Telegram update: значение `TELEGRAM_WEBHOOK_SECRET` в env не совпадает с secret, установленным через `pnpm setup:telegram`; запустите setup повторно.
- `/start` отвечает, но tenant не появляется: проверьте `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, применённые миграции и логи Next.js.
- Кнопка открывает не тот URL: проверьте `APP_BASE_URL`; итоговая ссылка должна быть `${APP_BASE_URL}/app`.
