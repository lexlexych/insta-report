# Ручной тест T-020 · Карточка черновика в Telegram + замещение старого

## Что сделано
- Добавлена HTML-карточка черновика для Telegram: клиентское сообщение, метка и текст ответа экранируются и обрезаются под лимит Telegram.
- Доставка черновика теперь удаляет предыдущую pending-карточку той же беседы, отправляет новую карточку и записывает её в `drafts` с UUID, созданным приложением.
- Гонка двух вебхуков обрабатывается безопасно: если БД уже содержит pending-черновик, только что отправленное сообщение удаляется, а пайплайн не падает.

## Подготовка
- Ветка: `git fetch && git checkout ticket/T-020 && pnpm install`.
- Env для demo-скрипта не нужен: `scripts/manual/T-020-demo.ts` сам выставляет безопасные dummy-значения и не ходит во внешние сервисы.
- Для интеграционной проверки в Telegram нужны заполненные переменные из `.env.example`: `TELEGRAM_BOT_TOKEN` (BotFather), `TELEGRAM_WEBHOOK_SECRET`, `MINIAPP_JWT_SECRET`, `ENCRYPTION_KEY`, `SUPABASE_URL`, `SUPABASE_SECRET_KEY`, `LLM_BASE_URL`, `LLM_API_KEY`, `LLM_MODEL_CLASSIFY`, `LLM_MODEL_DRAFT`, `APP_BASE_URL`, `CRON_SECRET`. Для БД нужен dev-проект Supabase с применёнными миграциями: `pnpm db:push`.

## Проверки
1. Запустите гигиенические проверки: `pnpm lint && pnpm typecheck` → обе команды завершаются без ошибок.
2. Запустите автономный demo-скрипт: `pnpm tsx scripts/manual/T-020-demo.ts` → все строки начинаются с `OK`, финальная строка `OK T-020 demo completed`.
3. В выводе demo-скрипта проверьте строку `OK HTML клиента экранирован` → критерий «HTML-инъекции невозможны» покрыт на строке с `<script>alert(1)</script>`.
4. В выводе demo-скрипта проверьте строку `OK старый pending отменён и удалён до отправки новой карточки` → порядок должен быть `cancel-old>delete-41>send-new>insert-...`, что демонстрирует замещение предыдущей живой карточки.
5. В выводе demo-скрипта проверьте строку `OK PendingExistsError удаляет только что отправленную карточку и не пробрасывается` → гонка уникального pending-индекса не оставляет лишнее сообщение в Telegram.
6. Интеграционно: поднимите приложение `pnpm dev`, настройте Instagram webhook как в предыдущих тикетах и отправьте клиентское сообщение в Direct → владелец получает Telegram-карточку с кнопками `🔗 Открыть в Instagram` и `✅ Отправить`, а в таблице `drafts` появляется одна запись `status=pending` с `tg_chat_id` и `tg_message_id`.
7. Отправьте второе клиентское сообщение в ту же Instagram-беседу до отправки первого черновика → старая карточка исчезает из Telegram-чата, новая остаётся, а в `drafts` для этой пары `tenant_id + conversation_key` есть только один `status=pending`.

## Негативные проверки
1. Временно очистите `tenants.tg_chat_id` у тестового тенанта и отправьте входящее Direct-сообщение → пайплайн логирует ошибку про отсутствие `tg_chat_id`, карточка не отправляется и приложение не падает.
2. Используйте demo-скрипт для XSS-строки `<script>alert(1)</script>` → в HTML карточки видны только `&lt;script&gt;...`, сырого `<script>` нет.

## Как понять, что тикет НЕ работает
- В Telegram остаются две карточки для одной беседы: проверьте логи вокруг `cancelPendingByConversation`, `deleteMessageSafe`, `sendMessageHTML` и уникальный индекс `drafts_pending_uniq` в Supabase.
- Telegram возвращает ошибку `message is too long`: проверьте обрезку `pendingText`/`draftText` и длину результата `renderDraftCard`.
- В карточке видны сырые HTML-теги из сообщения клиента: проверьте, что все пользовательские строки идут через `escapeHTML`.
- В `drafts` нет `tg_message_id`: проверьте ответ Telegram `sendMessage` и обработку `message_id` в доставке.
