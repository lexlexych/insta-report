# Ручной тест T-048 · Zernio: глобальный вебхук и события аккаунта

## Что сделано

Добавлен единый endpoint `POST /api/wh/zernio` для событий всего аккаунта Zernio. Он сверяет
HMAC-подпись от неизменённого тела запроса, немедленно подтверждает доставку и асинхронно
обновляет подключение Instagram. Скрипт настройки создаёт либо обновляет единственный webhook
`insta-reply`, не выводя его секрет.

## Подготовка

- ветка: `git fetch && git checkout zernio && pnpm install`
- env: заполните обязательные переменные из `.env.example`, а также `ZERNIO_API_KEY` и
  `ZERNIO_WEBHOOK_SECRET`. Для локальной проверки задайте `APP_BASE_URL` как публичный HTTPS-адрес
  туннеля; не вставляйте секрет в терминал, документацию или скриншоты.
- сервисы: примените миграции T-046 командой `pnpm db:push`, запустите `pnpm dev` и направьте
  публичный HTTPS-туннель (например, cloudflared) на порт 3000. После изменения `APP_BASE_URL`
  перезапустите dev-сервер.

## Проверки

1. Выполните `pnpm tsx scripts/manual/zernio-setup-webhook.ts` → в stdout напечатана настройка
   `insta-reply` с URL `<APP_BASE_URL>/api/wh/zernio`, четырьмя событиями и `isActive: true`.
   В выводе нет поля `secret` и его значения.
2. Запустите ту же команду второй раз → настройка обновлена с тем же `id`, не создаётся второй
   webhook. Если после ошибок доставки настройка была выключена, повторный запуск возвращает
   `isActive: true`.
3. В Zernio Dashboard вызовите `POST /v1/webhooks/test` для `insta-reply` → доставка получает
   HTTP `200` быстрее пяти секунд. Неведомое тестовое событие допускается: в серверном логе оно
   помечено как ignored, а Zernio не планирует повторную доставку.
4. Пройдите подключение из T-047 для тестового тенанта. В логах должен появиться
   `account.connected`; строка этого тенанта в `zernio_accounts` остаётся `active`, содержит
   текущий `zernio_account_id` и username. Повторная доставка не создаёт новую строку.
5. Отключите аккаунт в Zernio так, чтобы пришло `account.disconnected` с
   `disconnectionType: unintentional` → запись тенанта становится `error`, а владелец получает в
   Telegram сообщение о необходимости переподключения. Повторная доставка не создаёт повторных
   записей и не меняет аккаунт другого тенанта.
6. Для проверки подписи без Dashboard используйте PowerShell ниже с публичным URL вместо
   `<APP_BASE_URL>`; ответ должен содержать `StatusCode: 200`. Тело и подпись создаются в памяти,
   значение `ZERNIO_WEBHOOK_SECRET` не выводится.

   ```powershell
   $body = '{"id":"manual-t048","event":"unknown.event","timestamp":"2026-07-16T12:00:00Z"}'
   $key = [Text.Encoding]::UTF8.GetBytes($env:ZERNIO_WEBHOOK_SECRET)
   $hmac = [Security.Cryptography.HMACSHA256]::new($key)
   $signature = -join ($hmac.ComputeHash([Text.Encoding]::UTF8.GetBytes($body)) | ForEach-Object { $_.ToString('x2') })
   Invoke-WebRequest -Method Post -Uri '<APP_BASE_URL>/api/wh/zernio' -ContentType 'application/json' -Headers @{ 'X-Zernio-Signature' = $signature } -Body $body
   ```

## Негативные проверки

1. Повторите шаг 6 без заголовка `X-Zernio-Signature` → HTTP `401 Unauthorized`.
2. Повторите шаг 6, заменив один символ `$signature` перед запросом → HTTP `401 Unauthorized`.
3. Откройте `<APP_BASE_URL>/api/wh/zernio` методом `GET` → HTTP `405` либо `404`.
4. Временно очистите обе переменные `ZERNIO_API_KEY` и `ZERNIO_WEBHOOK_SECRET`, перезапустите
   сервер и выполните POST → HTTP `404`; затем верните обе переменные и повторите setup-скрипт.

## Как понять, что тикет НЕ работает

Если Zernio показывает retry или не получает `200`, проверьте публичность `APP_BASE_URL`, URL
настройки `insta-reply` и серверный лог `[wh/zernio]`. Для `401` проверьте, что webhook создан
именно setup-скриптом и что тело не меняется прокси до проверки подписи. Если статус аккаунта не
обновился, сопоставьте `account.profileId` или `account.accountId` из лога доставки с записью
`zernio_accounts`; неизвестные аккаунты намеренно только логируются.
