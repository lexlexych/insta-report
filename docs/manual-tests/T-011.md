# Ручной тест T-011 · Per-tenant вебхук Meta: verify + приём

## Что сделано

Добавлен роут `app/api/wh/ig/[tenantId]/route.ts`, обрабатывающий вебхук Meta для конкретного
тенанта: `GET` подтверждает подписку (verify handshake) и фиксирует факт хэндшейка
(`ig_connections.handshake_at`), `POST` проверяет подпись `x-hub-signature-256` по сырому телу
(timing-safe) и, если она валидна, немедленно отвечает `200 OK`, передавая обработку события в
`waitUntil(handleIgEvent(...))` (заглушка до T-016). Обработка не блокирует ответ Meta.

## Подготовка

- Ветка: `git fetch && git checkout ticket/T-011 && pnpm install`.
- Env: заполните переменные из `.env.example`, нужные для запуска сервера и Supabase
  (`SUPABASE_URL`, `SUPABASE_SECRET_KEY`, `ENCRYPTION_KEY` — сгенерировать `pnpm gen:key`).
  Секреты в логах не печатать.
- БД: примените миграции (включая новую `0003_handshake.sql`) на dev-проекте Supabase:
  `pnpm db:push`.
- Тестовые данные: создайте тенанта и подключение Instagram с ИЗВЕСТНЫМИ вам plaintext-значениями
  `verify_token` и `app_secret` (например, через Supabase SQL Editor или существующий флоу
  подключения Instagram в Mini App). Если создаёте вручную SQL — учтите, что `access_token_enc`/
  `app_secret_enc` должны быть значениями, зашифрованными модулем `src/lib/crypto.ts` (проще всего
  временно вызвать `encrypt()` через `pnpm tsx` или создать подключение через существующий UI/API,
  если он уже пишет `app_secret_enc`); `verify_token` хранится в открытом виде (см. схему в
  `docs/plan.md §2.4`).
- Сервис: запустите `pnpm dev` (порт 3000 по умолчанию).

## Проверки (каждая: Действие → Ожидаемо)

Запустите автоматизированный прогон демо-скрипта (покрывает все проверки ниже одной командой):

```bash
WH_BASE_URL=http://localhost:3000 \
WH_TENANT_ID=<uuid тенанта> \
WH_VERIFY_TOKEN=<plaintext verify_token> \
WH_APP_SECRET=<plaintext app_secret> \
pnpm tsx scripts/manual/T-011-demo.ts
```

Ожидаемо: все 7 строк выводятся с `OK:` и скрипт завершается сообщением
`T-011 demo completed: все проверки прошли.`

Также при желании проверьте вручную:

1. `GET /api/wh/ig/<tenantId>?hub.mode=subscribe&hub.verify_token=<verify_token>&hub.challenge=abc123`.
   → Ожидаемо: HTTP 200, тело ответа — ровно `abc123`; в `ig_connections.handshake_at` тенанта
   проставлена свежая метка времени (`ig_connections.status` при этом НЕ меняется на `active`).
2. `POST /api/wh/ig/<tenantId>` с телом `{"object":"instagram","entry":[]}` и заголовком
   `x-hub-signature-256: sha256=<hex HMAC-SHA256(app_secret, тело)>`.
   → Ожидаемо: HTTP 200, тело `OK`; в `ig_connections.webhook_last_seen_at` тенанта — свежая метка
   времени; в логах сервера видно, что `handleIgEvent` был вызван (текущая заглушка ничего не
   делает, но не бросает исключений).

## Негативные проверки (см. также демо-скрипт, пункты 2/3/5/6/7)

1. `GET` с неверным `hub.verify_token` → HTTP 403, `handshake_at` не обновляется.
2. `GET` с `tenantId`, для которого нет `ig_connections` → HTTP 404.
3. `POST` без заголовка `x-hub-signature-256` → HTTP 401, `handleIgEvent` не вызывается.
4. `POST` с подписью, посчитанной по другому секрету → HTTP 401.
5. `POST` с валидной подписью, но телом, изменённым после подписания → HTTP 401 (подпись больше не
   соответствует телу).

## Как понять, что тикет НЕ работает

- Демо-скрипт падает с `WH FAILED` или строкой `FAIL:` — смотрите, какой конкретно шаг не прошёл, и
  сверяйтесь с логами `pnpm dev` (сервер печатает `[wh/ig] invalid signature tenant=...` при 401,
  без утечки секрета/тела/подписи).
- Если `GET` возвращает не сам `challenge`, а обёрнутый JSON — проверьте, что роут отдаёт
  `new Response(challenge, ...)`, а не `Response.json(...)`.
- Если `ig_connections.status` меняется на `active` после хэндшейка — это регрессия (по тикету
  меняться должен только `handshake_at`, не `status`).
- Если ответ на `POST` задерживается (Meta видит таймаут/ретраит) — проверьте, что обработка вызвана
  через `waitUntil(...)`, а не `await` перед ответом.
- Если после невалидной подписи всё равно виден вызов `handleIgEvent` — проверьте порядок проверок в
  `route.ts` (подпись должна проверяться раньше, чем происходит `JSON.parse`/`waitUntil`).
