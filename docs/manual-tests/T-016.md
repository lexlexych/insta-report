# Ручной тест T-016 · Парсер событий Instagram и dedup

## Что сделано
- Сырое тело Meta-вебхука Instagram превращается в типизированное событие `IgEvent` (`src/lib/pipeline/parse.ts`, тип в `src/lib/pipeline/types.ts`): Zod-схема с `.passthrough()`, правило сторон echo/incoming портировано из n8n-ноды «Parse & Route».
- `src/lib/pipeline/handleIgEvent.ts` (замена заглушки T-011): parse → dedup по `mid` через `processedEvents.tryInsert` → диспатч echo (`handleEcho`, заглушка до T-022) / входящее (`handleIncoming`, заглушка до T-017+). Мусор и read-события — мягкий выход без исключения.
- Для библиотечного модуля без UI добавлен demo-скрипт `scripts/manual/T-016-demo.ts` и 4 реалистичные фикстуры `tests/fixtures/ig/*.json`, покрывающие edge cases из раздела «Тесты» тикета (по OVERRIDES §1 вместо vitest).

## Подготовка
- Ветка: `git fetch && git checkout claude/ticket-t-016-3xnedj && pnpm install`.
- Env: не нужны. Demo-скрипт не обращается к БД, сети или `process.env` — обработчики (`tryInsert`/`handleEcho`/`handleIncoming`) внедряются как шпионы, а `parseIgEvent` — чистая функция.
- Сервисы: ничего поднимать не нужно.

## Проверки (Действие → Ожидаемо)
1. `pnpm lint` → завершается без ошибок ESLint.
2. `pnpm typecheck` → TypeScript завершается без ошибок.
3. `pnpm tsx scripts/manual/T-016-demo.ts` → печатается серия строк `OK: …` и финальная `T-016 demo completed: все проверки прошли.` (exit code 0). Ни одной строки `FAIL:`.
4. **AC «Повторная доставка того же mid не обрабатывается»** — в выводе demo сценарий «дубликат mid»: строки `дубликат mid -> tryInsert вызван 1 раз`, `дубликат mid -> handleIncoming не вызван`, `дубликат mid -> handleEcho не вызван` со статусом `OK`. То есть при `tryInsert=false` (mid уже в `processed_events`) ни один обработчик не запускается.
5. **AC «Правило сторон echo/incoming покрыто»** — строки `incoming_text -> accountId=business` / `contactId=client` и `echo -> accountId=sender (стороны развёрнуты)` / `contactId=recipient (стороны развёрнуты)` со статусом `OK`. Подтверждает: для входящего `accountId=recipient.id`, для echo `accountId=sender.id` (стороны меняются местами).
6. **AC «Фикстуры лежат в tests/fixtures/ig и используются далее»** — `ls tests/fixtures/ig` показывает `incoming_text.json`, `echo.json`, `attachment_only.json`, `read_event.json`; demo грузит их именно оттуда (`loadFixture(...)`) и печатает по ним строки `OK`.
7. Вложение без текста: строки `attachment_only -> text=""` и `attachment_only -> hasAttachments=true` со статусом `OK`.
8. Архитектурная граница: `rg "process\.env" src/lib/pipeline` → нет совпадений (env не читается вне `src/lib/env.ts`). `rg "from '@/lib/db'" src/lib/pipeline` → единственное совпадение — динамический `await import('@/lib/db')` внутри `handleIgEvent.ts` (статический импорт БД отсутствует, иначе demo под `tsx` упал бы на `server-only`).

## Негативные проверки
1. Мусорные/нерелевантные тела → `parseIgEvent` возвращает `null` (в demo строки `OK`): `{}`, `null`, строка `'nope'`, `{ object: 'page', entry: [] }`, `{ object: 'instagram', entry: [{ id: 'x', time: 1, messaging: [] }] }`.
2. Read-событие (в теле нет `message`, только `read`) → `read_event -> null (нет message)` со статусом `OK`: квитанции о прочтении не порождают обработку.
3. Мусорное тело в `handleIgEvent`: строки `мусор -> tryInsert не вызван`, `мусор -> handleIncoming не вызван`, `мусор -> handleEcho не вызван` — до dedup и обработчиков дело не доходит, исключение не бросается (мягкий return).
4. Пустое входящее (есть `mid`, но нет ни текста, ни вложений) после успешного `tryInsert` → `пустое входящее -> handleIncoming не вызван` / `handleEcho не вызван` со статусом `OK` (срабатывает гуард `!ev.text && !ev.hasAttachments`).

## Как понять, что тикет НЕ работает
- `pnpm tsx scripts/manual/T-016-demo.ts` печатает любую строку `FAIL:` или завершается с ненулевым кодом — смотреть, какой именно инвариант нарушен (метка строки указывает сценарий).
- Строка вроде `Cannot import module ... server-only` при запуске demo — значит в `handleIgEvent.ts`/`parse.ts` появился статический импорт `@/lib/db` (или транзитивно `db/client.ts`); БД должна подключаться только динамически внутри `handleIgEvent`.
- Демо падает на `echo -> accountId=sender` — перепутано правило сторон в `parseIgEvent` (echo должен давать `accountId=sender.id`).
- Демо падает на `дубликат mid -> handleIncoming не вызван` — dedup не блокирует повторную доставку: проверить порядок `tryInsert` перед диспатчем в `handleIgEvent.ts`.
- `pnpm typecheck` краснеет на `entry[0]`/`messaging[0]` — потеряны гуарды под `noUncheckedIndexedAccess` в `parse.ts`.
