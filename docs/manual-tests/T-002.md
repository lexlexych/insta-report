# Ручной тест T-002 · Схема БД и миграции Supabase

## Что сделано

Добавлены SQL-миграции полной схемы БД (`supabase/migrations/0001_init.sql`,
`0002_increment_usage.sql`): 7 таблиц с индексами и RLS deny-all для anon, функции
`seed_default_labels` и `increment_usage`. Добавлены скрипты `pnpm db:push` / `pnpm db:types`
и вручную написанный `src/lib/db/types.gen.ts` (до линковки живого проекта).

## Подготовка

- ветка: `git fetch && git checkout claude/ticket-t002-g884dc && pnpm install`
- нужен пустой (можно бесплатный) dev-проект на [supabase.com](https://supabase.com) —
  создайте новый проект в дашборде, если своего ещё нет.
- из настроек проекта возьмите: `Project Reference ID` (Settings → General),
  `Project URL` и `service_role` key (Settings → API) — впишите их в `.env.local`
  как `SUPABASE_URL` и `SUPABASE_SERVICE_ROLE_KEY` (см. `.env.example`).
- войдите в Supabase CLI и свяжите проект:
  ```bash
  pnpm dlx supabase login
  pnpm dlx supabase link --project-ref <ваш-project-ref>
  ```

## Проверки (каждая: Действие → Ожидаемо)

1. **Применение миграций.**
   Действие: `pnpm db:push` (это `supabase db push`).
   Ожидаемо: команда завершается без ошибок, сообщает о применении
   `0001_init.sql` и `0002_increment_usage.sql`.

2. **Таблицы созданы.**
   Действие: в дашборде Supabase → Table Editor (или `pnpm dlx supabase db diff` / SQL-редактор
   с `select table_name from information_schema.tables where table_schema='public';`).
   Ожидаемо: присутствуют ровно 7 таблиц: `tenants`, `ig_connections`, `labels`, `drafts`,
   `processed_events`, `message_log`, `usage_stats`.

3. **Ключевые индексы созданы.**
   Действие: SQL-редактор →
   ```sql
   select indexname from pg_indexes where schemaname='public'
   order by indexname;
   ```
   Ожидаемо: в списке есть `drafts_pending_uniq`, `processed_events_uniq`, `usage_stats_uniq`,
   плюс безымянные индексы на `drafts(tenant_id, status)` и `message_log(tenant_id, created_at)`.

4. **RLS включён везде.**
   Действие: SQL-редактор →
   ```sql
   select relname, relrowsecurity from pg_class
   where relname in ('tenants','ig_connections','labels','drafts',
                      'processed_events','message_log','usage_stats');
   ```
   Ожидаемо: `relrowsecurity = true` для всех 7 строк, политик (Policies) в Table Editor нет ни
   у одной таблицы.

5. **Функции созданы и работают.**
   Действие: SQL-редактор →
   ```sql
   insert into tenants (telegram_user_id) values (111) returning id;
   -- подставьте полученный id вместо <tenant_id> ниже
   select seed_default_labels('<tenant_id>');
   select name, sort from labels where tenant_id = '<tenant_id>';
   ```
   Ожидаемо: `seed_default_labels` выполняется без ошибок; в `labels` появляется одна строка
   `Без категории` / `sort = 999`.

6. **Инкремент статистики атомарный (upsert).**
   Действие: там же —
   ```sql
   select increment_usage('<tenant_id>', current_date, 1, 100, 50, 1, 0);
   select increment_usage('<tenant_id>', current_date, 1, 100, 50, 0, 1);
   select * from usage_stats where tenant_id = '<tenant_id>';
   ```
   Ожидаемо: одна строка на `(tenant_id, day)` с суммарными значениями
   `llm_calls=2, tokens_in=200, tokens_out=100, drafts_created=1, drafts_sent=1`.

7. **Типы компилируются.**
   Действие: `pnpm typecheck`.
   Ожидаемо: завершается без ошибок (в т.ч. использует `src/lib/db/types.gen.ts`).

## Негативные проверки

1. **Deny-all для anon.**
   Действие: в SQL-редакторе выполнить запрос от имени роли `anon`
   (или через `supabase-js` с `anon`-ключом вместо service role) —
   ```sql
   set role anon;
   select * from tenants;
   ```
   Ожидаемо: 0 строк (RLS блокирует доступ — не ошибка авторизации, а пустой результат/deny),
   в отличие от запроса с `service_role`, который вернёт добавленную ранее запись.

2. **Дубликат pending-черновика запрещён.**
   Действие:
   ```sql
   insert into drafts (tenant_id, conversation_key, status)
   values ('<tenant_id>', 'acc1:contact1', 'pending');
   insert into drafts (tenant_id, conversation_key, status)
   values ('<tenant_id>', 'acc1:contact1', 'pending');
   ```
   Ожидаемо: второй `insert` падает с ошибкой нарушения уникального индекса
   `drafts_pending_uniq` (`duplicate key value violates unique constraint`).

3. **Повторный сид меток не дублирует.**
   Действие: повторно выполнить `select seed_default_labels('<tenant_id>');` из проверки 5.
   Ожидаемо: выполняется без ошибок, но новой строки в `labels` не добавляется
   (`on conflict do nothing` + `unique(tenant_id, name)`).

## Как понять, что тикет НЕ работает

- `pnpm db:push` падает с синтаксической ошибкой SQL → смотреть текст ошибки, обычно указывает
  строку/конструкцию в `supabase/migrations/*.sql`.
- Таблицы создались, но RLS выключен (`relrowsecurity = false`) → запрос из проверки 4 покажет
  это явно; значит потерялась строка `alter table ... enable row level security;`.
- `select * from tenants;` под `anon`-ролью возвращает данные (не 0 строк) → RLS не работает,
  критическая проблема безопасности, тикет не принимать.
- `pnpm typecheck` падает на `src/lib/db/types.gen.ts` → несоответствие вручную написанных типов
  реальной схеме миграций.
