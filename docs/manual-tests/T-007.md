# Ручной тест T-007 · Аутентификация Mini App (initData → JWT)

## Что сделано
- Сервер проверяет Telegram `initData` по алгоритму Web App, включая `hash`, `auth_date` и timing-safe сравнение.
- `/api/miniapp/auth` создаёт/обновляет tenant по Telegram user id, подписывает JWT-сессию и кладёт её в httpOnly cookie `session`.
- Mini App при старте автоматически отправляет raw initData на `/api/miniapp/auth`; при ошибке показывает экран с кнопкой «Повторить».

## Подготовка
- Ветка: `git fetch && git checkout ticket/T-007 && pnpm install`.
- Если `pnpm install` ранее не смог скачать `jose` из registry, повторите установку в сети с доступом к npm registry: зависимость добавлена в `package.json`.
- Env в `.env.local`:
  - `TELEGRAM_BOT_TOKEN` — токен центрального бота из BotFather.
  - `MINIAPP_JWT_SECRET` — длинная случайная строка для подписи cookie-сессий.
  - Supabase-переменные из предыдущих тикетов: `SUPABASE_URL`, `SUPABASE_SECRET_KEY`; примените миграции через `pnpm db:push` на dev-проекте.
  - Остальные обязательные переменные из `src/lib/env.ts` должны быть заполнены валидными тестовыми значениями, чтобы приложение стартовало.
- Сервисы: запустите `pnpm dev`; для проверки внутри Telegram используйте публичный HTTPS URL (например, туннель) как Web App URL бота.

## Проверки
1. Откройте Mini App из Telegram-бота.
   → Экран не должен зависнуть на пустой загрузке; после автоматического POST `/api/miniapp/auth` отображается Mini App shell.
2. В DevTools/Network найдите запрос `POST /api/miniapp/auth`.
   → Статус `200`, тело содержит `{"ok":true,"tenant":{"id":...}}`; в ответе есть `Set-Cookie` с `session=...; HttpOnly; Secure; SameSite=None; Path=/; Max-Age=43200`.
3. В Supabase Table Editor откройте таблицу `tenants` и найдите запись с вашим Telegram `user.id`.
   → Запись существует; `telegram_user_id` равен Telegram user id, `tg_chat_id` равен тому же id.
4. Обновите Mini App внутри Telegram.
   → Аутентификация повторяется без ручного ввода; пользователь снова попадает в приложение.
5. Критерий `/api/miniapp/*`: на момент T-007 единственный route под `/api/miniapp/` — `/api/miniapp/auth`, поэтому исключение для auth покрывает весь текущий набор роутов; следующие тикеты должны использовать `requireTenant` через `apiHandler` для новых `/api/miniapp/*` роутов.

## Негативные проверки
1. Отправьте запрос с пустым телом:
   ```bash
   curl -i -X POST http://localhost:3000/api/miniapp/auth \
     -H 'Content-Type: application/json' \
     -d '{}'
   ```
   → Ответ `400` с `{"ok":false,"error":"malformed"}`.
2. Скопируйте реальный payload `initData` из успешного запроса, измените один символ в JSON `user`, но оставьте прежний `hash`, и отправьте повторно через curl.
   → Ответ `401` с `{"ok":false,"error":"bad_hash"}`.
3. Повторите запрос с initData, у которого отсутствует параметр `hash`.
   → Ответ `401` с `{"ok":false,"error":"malformed"}`.
4. Для проверки `expired` временно отправьте валидно подписанную фикстуру с `auth_date` старше 3600 секунд (можно сгенерировать локальным одноразовым скриптом по алгоритму Telegram из тикета).
   → Ответ `401` с `{"ok":false,"error":"expired"}`.
5. Откройте страницу Mini App не из Telegram WebView, а напрямую в браузере.
   → Показывается fallback «Öffne die App über den Telegram-Bot …»; при недоступном/битом initData внутри Telegram должен появиться экран ошибки с кнопкой «Wiederholen / Повторить».

## Как понять, что тикет НЕ работает
- `POST /api/miniapp/auth` возвращает `500`: проверьте обязательные env-переменные и доступ к Supabase; детали будут в серверном логе Next.js.
- Cookie не сохраняется в Telegram WebView: проверьте, что URL HTTPS и cookie содержит `Secure; SameSite=None`.
- Ответ `bad_hash` на реальный Telegram initData: проверьте, что `TELEGRAM_BOT_TOKEN` соответствует именно тому боту, из которого открывается Mini App.
- Tenant не появляется в Supabase: проверьте `SUPABASE_URL`, `SUPABASE_SECRET_KEY` и что миграции T-002 применены.
