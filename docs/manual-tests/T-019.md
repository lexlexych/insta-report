# Ручной тест T-019 · Генерация черновика ответа

## Что сделано
- Добавлена генерация черновика ответа из системного промпта тенанта, базы знаний, истории диалога, неотвеченного текста клиента и инструкции выбранной категории.
- Добавлен безопасный fast-path для сообщений «только вложение»: LLM не вызывается, возвращается шаблонный уточняющий ответ на русском или немецком.
- Собран частичный orchestrator входящего сообщения: активное Instagram-подключение → контекст → классификация → черновик → заглушка доставки T-020 → лог входящего сообщения и инкремент статистики.

## Подготовка
- Ветка: `git fetch && git checkout ticket/T-019 && pnpm install`.
- Env: для локального demo-скрипта реальные секреты не нужны — скрипт сам выставляет dummy-значения переменных из `.env.example` и мокает LLM-вызовы.
- Сервисы: Supabase, Meta, Telegram и LLM-провайдер для этой проверки поднимать не нужно; тикет проверяется через одноразовый локальный скрипт и статические проверки.

## Проверки
1. Выполните `pnpm tsx scripts/manual/T-019-demo.ts`.
   - Ожидаемо: все строки начинаются с `OK`, в конце напечатано `OK T-019 demo completed`.
   - Покрывает: prompt содержит базу знаний, историю, pendingText и инструкцию метки; обычный сценарий вызывает LLM с `temperature=0.7` и `maxTokens=500`; usage возвращается наружу; обрамляющие кавычки у ответа срезаются.
2. В выводе demo-скрипта найдите строки `OK вложение без текста возвращает шаблон` и `OK вложение без текста не тратит usage`.
   - Ожидаемо: обе строки присутствуют.
   - Покрывает: кейс «вложение без текста» не тратит LLM-вызов и отдаёт шаблонный уточняющий текст.
3. В выводе demo-скрипта найдите строку `OK пустой ответ LLM даёт LlmError empty_draft`.
   - Ожидаемо: строка присутствует.
   - Покрывает: пустой LLM-ответ после trim считается ошибкой `LlmError('empty_draft')`.
4. Выполните `pnpm lint && pnpm typecheck`.
   - Ожидаемо: обе команды завершаются без ошибок.
   - Покрывает: обязательный Definition of Done проекта без автоматических тестов.

## Негативные проверки
1. Временно измените в `scripts/manual/T-019-demo.ts` мок обычного LLM-ответа так, чтобы он возвращал `text: '   '`, и запустите `pnpm tsx scripts/manual/T-019-demo.ts`.
   - Ожидаемо: сценарий пустого ответа падает в `LlmError empty_draft`; после проверки откатите изменение.
2. Временно измените expected-ключ в проверке `conversationKey(ev) === 'ig-business:ig-client'` на неверный текст и запустите `pnpm tsx scripts/manual/T-019-demo.ts`.
   - Ожидаемо: скрипт печатает `FAIL conversationKey = accountId:contactId` и завершается ошибкой; после проверки откатите изменение.

## Как понять, что тикет НЕ работает
- Demo-скрипт печатает `FAIL ...`: смотрите конкретную строку — она указывает, какой критерий сломан (prompt, attachment-only, empty draft или key).
- `Cannot find module 'server-only'` в demo-скрипте означает, что в проверяемый путь попал прямой runtime-импорт серверного DB/LLM-модуля; генерация с моками должна обходиться без него.
- `pnpm typecheck` ругается на типы `Tenant`, `Label`, `ConversationContext` или `IgEvent`: проверьте совместимость `src/lib/pipeline/draft.ts` и `src/lib/pipeline/handleIncoming.ts` с типами БД и pipeline.
