# Ручной тест T-036 · Реестр Instagram-аккаунтов и сфера деятельности тенанта

## Что сделано

Добавлен реестр аккаунтов Instagram, вручную подключённых к общей Meta App. Заявка хранит
нормализованный username, статус подтверждения и сведения об администраторе, который её подтвердил.
У тенанта появилось необязательное поле сферы деятельности.

## Подготовка

- ветка: `git fetch && git checkout ticket/T-036 && pnpm install`
- env: заполните `SUPABASE_URL` и `SUPABASE_SECRET_KEY` в локальном `.env` данными dev-проекта
  Supabase. URL и service-role key возьмите в Supabase Dashboard: **Project Settings → API**.
- сервисы: используйте отдельный dev-проект Supabase и выполните `pnpm db:push`. Для demo-скрипта
  нужен UUID уже существующего тенанта из таблицы `tenants`.

## Проверки

1. Выполните `pnpm db:push` на dev-проекте, где уже применена миграция `0010_remove_own_app.sql` →
   миграция применяется без ошибок.
2. В Supabase Dashboard откройте **Table Editor** → `ig_accounts` → таблица существует с полями
   `id`, `ig_username`, `tenant_id`, `status`, `requested_at`, `approved_at`, `approved_by_tg_id`;
   `id`, `requested_at` и `status` имеют значения по умолчанию, а `tenant_id`, `approved_at` и
   `approved_by_tg_id` допускают `NULL`.
3. В SQL Editor выполните `select column_name from information_schema.columns where table_name = 'tenants' and column_name = 'business_sphere';` → возвращается одна строка `business_sphere`;
   поле допускает `NULL`.
4. В SQL Editor откройте определение `ig_accounts` → `ig_username` имеет ограничение уникальности,
   `status` допускает только `pending` и `approved`, а внешний ключ `tenant_id` при удалении тенанта
   устанавливает `NULL`.
5. В SQL Editor выполните `select relrowsecurity from pg_class where oid = 'ig_accounts'::regclass;` →
   возвращается `true`; в Dashboard у таблицы нет RLS policies.
6. Скопируйте UUID существующего тенанта и выполните `pnpm tsx scripts/manual/T-036-demo.ts <tenant-id>` →
   выводится `T-036 demo OK`. Скрипт проверяет нормализацию ` @Foo.Bar ` в `foo.bar`, создание
   pending-заявки, единственное успешное подтверждение, а также привязку approved-заявки без тенанта.

## Негативные проверки

1. Временно измените в demo-скрипте входной username на кириллицу, значение с пробелом или строку из
   31 символа и запустите его → проверка завершается ошибкой, а некорректная заявка не появляется в БД.
2. В SQL Editor попробуйте вручную вставить вторую строку с тем же `ig_username` → PostgreSQL сообщает
   об ошибке уникальности, дубликат не создаётся.

## Как понять, что тикет НЕ работает

Если `pnpm db:push` сообщает об ошибке, убедитесь, что миграция `0010_remove_own_app.sql` уже есть в
истории dev-проекта, затем посмотрите текст ошибки в терминале. Если demo-скрипт не проходит,
проверьте корректность UUID тенанта, service-role key и строки `ig_accounts` в Table Editor. Отсутствие
RLS, policy у таблицы или возможность создать дубликат username означает, что миграция применена не
полностью.
