# Ручной тест T-039 · Демонтаж LLM-генерации онбординга

## Что сделано

Онбординг больше не создаёт базу знаний через LLM из свободного описания бизнеса. Новый пользователь выбирает сферу и редактирует готовый шаблон базы знаний.

Удалены старый API-маршрут и связанные с ним переводы. Миграция удаляет из таблицы tenant устаревшую колонку с описанием бизнеса; существующие базы знаний и системные промпты она не меняет.

## Подготовка

- Ветка: `git fetch && git checkout ticket/T-039 && pnpm install`.
- env: заполните значения из `.env.example`, нужные для Mini App, Supabase, Telegram-бота и LLM. Для проверки онбординга LLM-ключ не используется; токен бота создаётся в BotFather.
- Сервисы: до запуска сделайте резервную копию свободных описаний бизнеса, если они нужны: после миграции их нельзя восстановить. Примените миграции к dev-проекту Supabase командой `pnpm db:push`, затем запустите `pnpm dev` и откройте Mini App из Telegram.

## Проверки

1. Выполните `pnpm db:push`, затем в SQL Editor dev-проекта выполните `select column_name from information_schema.columns where table_schema = 'public' and table_name = 'tenants' and column_name = 'org_description';` → запрос не возвращает строк. У существующего tenant проверьте `knowledge_base` и `system_prompt` → их значения сохранены.
2. Откройте Mini App новым пользователем и пройдите онбординг: выберите сферу, укажите название и Instagram-аккаунт, перейдите через заявку или подключение к шаблону базы знаний, отредактируйте и завершите → визард проходит до дашборда без экрана свободного описания и без ожидания генерации.
3. Во время нового прохода откройте DevTools → Network и отфильтруйте `generate` → запросов к `/api/miniapp/onboarding/generate` нет; на шаге базы знаний отправляется только завершение онбординга.
4. После завершения откройте «Настройки» → база знаний и системный промпт доступны для редактирования. Откройте «Тест-чат» и отправьте вопрос → появляется ответ-черновик по сохранённой базе знаний.
5. Для tenant с незавершённым онбордингом откройте «Тест-чат» → сервер возвращает ожидаемую ошибку `409` с кодом `onboarding_required`; после завершения онбординга эта ошибка не возникает. Откройте дашборд → он загружается без изменения поведения.

## Негативные проверки

1. После запуска приложения вручную отправьте `POST` на `/api/miniapp/onboarding/generate` с корректной авторизацией → маршрут отсутствует (404), а данные tenant и база знаний не изменяются.
2. Перед `pnpm db:push` поместите в dev-базу тестовое значение `org_description`, затем примените миграцию → колонка и её тестовое значение удалены. Выполняйте это только на dev-данных: удаление необратимо.

## Как понять, что тикет НЕ работает

Если в онбординге есть textarea со свободным описанием или экран «генерации», проверьте исходящие запросы во вкладке Network: обращения к `/api/miniapp/onboarding/generate` означают регрессию. Если миграция не применена, запрос к `information_schema.columns` вернёт `org_description`; проверьте вывод `pnpm db:push`. Если настройки, симулятор или дашборд не открываются, проверьте их запросы Network и логи приложения, а также наличие `knowledge_base` и `system_prompt` у tenant в Supabase.
