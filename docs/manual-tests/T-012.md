# Ручной тест T-012 · Визард подключения Instagram (UI)

## Что сделано (2-4 строки, по-человечески)
Добавлена страница `/app/connect-instagram` — пошаговый визард-аккордеон, который проводит владельца
бизнеса через создание своего Meta App и выдаёт ему наши webhook-реквизиты (Callback URL + Verify Token)
с копированием в один тап. Внизу форма для App Secret и Access Token: секреты уходят на бэкенд
(`/api/miniapp/ig/connect`), шифруются в БД и наружу больше не отдаются — при повторном входе поля
показаны маской `••••••••` с кнопкой «Заменить». Ссылка «Подключить Instagram» есть на дашборде и на
финальном шаге онбординга.

## Подготовка
- ветка: `git fetch && git checkout claude/t-013-build-uxnkct && pnpm install`
- env (`.env.local`, значения — из твоего dev-окружения):
  - `APP_BASE_URL` — публичный URL приложения (напр. `https://<твой-проект>.vercel.app`, либо адрес
    туннеля для локали). Именно из него строится `webhookUrl`, проверяем это ниже.
  - `ENCRYPTION_KEY` — 32 байта base64 (см. `scripts/gen-key.ts` из T-003). Нужен, чтобы репозиторий
    смог зашифровать введённые токены.
  - `SUPABASE_URL` / `SUPABASE_SECRET_KEY` — dev-проект Supabase с применёнными миграциями
    (`pnpm db:push`): нужна таблица `ig_connections`.
  - `TELEGRAM_BOT_TOKEN`, `TELEGRAM_WEBHOOK_SECRET`, `MINIAPP_JWT_SECRET` — чтобы пройти авторизацию
    Mini App (страница защищена `requireTenant`, нужна сессионная cookie от `/api/miniapp/auth`).
- сервисы: `pnpm dev`. Открывать Mini App нужно из Telegram (кнопка меню бота ведёт на `{APP_BASE_URL}/app`),
  т.к. для сессии требуется валидный `initData`. Быстрый путь без телефона: пройди `/start` у бота
  и открой Mini App с ПК-клиента Telegram, затем в браузере DevTools перейди на `/app/connect-instagram`
  в том же WebView (cookie сессии уже установлена авторизацией при старте Mini App).

## Проверки (каждая: Действие → Ожидаемо)

1. **Первый заход создаёт connection с verify_token** _(AC: webhookUrl/verifyToken корректны)_
   - Действие: авторизованным открой `/app/connect-instagram`. Открой DevTools → Network → ответ
     `GET /api/miniapp/ig/connect`.
   - Ожидаемо: ответ `200` с JSON вида
     `{ status:"pending", igUsername:null, webhookUrl:"…", verifyToken:"<32 hex-символа>",
     hasToken:false, hasSecret:false, webhookLastSeenAt:null, handshakeAt:null }`.
     В таблице `ig_connections` появилась строка для твоего tenant с непустым `verify_token`.

2. **webhookUrl построен верно** _(AC: webhookUrl корректен)_
   - Действие: сравни `webhookUrl` из ответа со значением `APP_BASE_URL`.
   - Ожидаемо: ровно `"{APP_BASE_URL}/api/wh/ig/{tenantId}"`. Этот же формат принимает боевой роут
     `app/api/wh/ig/[tenantId]/route.ts` (T-011) — т.е. URL, который человек впишет в Meta, рабочий.

3. **Секреты НЕ приходят с бэкенда** _(AC: секреты никогда не возвращаются)_
   - Действие: изучи тело ответа `GET /api/miniapp/ig/connect` (шаг 1) полностью.
   - Ожидаемо: полей `accessToken`/`appSecret`/`access_token_enc`/`app_secret_enc` НЕТ. Есть только
     булевы `hasToken`/`hasSecret`. Расшифрованных или зашифрованных значений секретов в ответе нет.

4. **Копирование в один тап** _(AC: копируются в один тап)_
   - Действие: раскрой шаг 5 «Настройте Webhooks», нажми кнопку копирования рядом с «Callback URL»,
     затем рядом с «Verify Token». Вставь скопированное в любое поле.
   - Ожидаемо: вставляется ровно `webhookUrl` и `verifyToken` соответственно; на кнопке на ~1.5 c
     появляется «Скопировано ✓», затем возвращается «Копировать».

5. **Аккордеон-шаги и плейсхолдеры скриншотов**
   - Действие: последовательно кликай заголовки шагов 1–6.
   - Ожидаемо: шаг раскрывается/сворачивается; в шагах со скриншотами (`step1.png`/`step2.png` физически
     нет) вместо картинки — оформленный плейсхолдер «Скриншот появится здесь», страница НЕ падает.

6. **Сохранение секретов (happy path)** _(AC: маскирование при повторном входе)_
   - Действие: в шаге 6 введи App Secret и Access Token (каждое ≥10 символов, любые тестовые строки,
     напр. `test_app_secret_123` и `test_access_token_123`), нажми «Показать»/«Скрыть» (глазок),
     затем «Сохранить». Смотри Network.
   - Ожидаемо: `POST /api/miniapp/ig/connect` → `200 {ok:true}`. В `ig_connections` поля
     `access_token_enc`/`app_secret_enc` заполнены и **не равны** введённому тексту (формат
     `v1:...:...:...`). Появляется блок диагностики-плейсхолдер «Проверка появится здесь» (T-014).

7. **Повторный вход — статус и маскированные поля** _(AC: повторный вход показывает статус и маску)_
   - Действие: обнови страницу `/app/connect-instagram`.
   - Ожидаемо: `GET` вернул `hasToken:true`, `hasSecret:true`; **тот же** `verifyToken`, что и в шаге 1
     (не перегенерирован). В форме App Secret и Access Token показаны как `••••••••` с кнопкой «Заменить»
     (не пустой инпут, не открытый текст). Нажатие «Заменить» открывает поле ввода заново.

8. **Секреты не пишутся в логи** _(AC: секреты не пишутся в логи)_
   - Действие: во время шага 6 смотри консоль сервера `pnpm dev`.
   - Ожидаемо: в stdout/stderr нет ни `test_app_secret_123`, ни `test_access_token_123`, ни расшифрованных
     значений. (При ошибке допускается лог объекта ошибки без тела запроса.)

9. **Ссылки на визард со стороны продукта**
   - Действие: открой дашборд `/app` и финальный шаг онбординга `/app/onboarding` (шаг «done»).
   - Ожидаемо: на обоих есть кнопка/ссылка «Подключить Instagram», ведущая на `/app/connect-instagram`.

## Негативные проверки (минимум одна: неверный ввод/подпись/токен → корректная ошибка)

1. **Пустая/частично пустая форма не отправляется** _(край: форма не шлётся с пустыми полями)_
   - Действие: в шаге 6 оставь оба поля пустыми (или введи <10 символов) и нажми «Сохранить».
   - Ожидаемо: клиентская валидация показывает «Минимум 10 символов», POST не уходит. Если всё же
     отправить тело без полей вручную (`curl -X POST .../api/miniapp/ig/connect -d '{}' -H 'Content-Type: application/json'`
     с валидной cookie) → `400 {ok:false,error:"malformed"}`.

2. **Доступ без сессии → 401** _(край: requireTenant)_
   - Действие: `curl -i {APP_BASE_URL}/api/miniapp/ig/connect` без cookie сессии.
   - Ожидаемо: `401` (роут защищён `requireTenant`), запись в БД не создаётся.

## Как понять, что тикет НЕ работает (типовые симптомы и куда смотреть)
- В ответе `GET` виден `accessToken`/`appSecret` или их расшифровка → утечка секрета, смотри
  `app/api/miniapp/ig/connect/route.ts` (в JSON должны быть только `hasToken`/`hasSecret`).
- `webhookUrl` не совпал с `{APP_BASE_URL}/api/wh/ig/{tenantId}` → проверь `APP_BASE_URL` в env и
  строку 37 роута; Meta не пройдёт verify, если URL кривой.
- Повторный вход показал пустые поля вместо `••••` → смотри инициализацию `editToken/editSecret`
  в `app/(miniapp)/app/connect-instagram/page.tsx` (должно зависеть от `hasToken/hasSecret`).
- `verifyToken` меняется при каждом обновлении страницы → GET ошибочно пересоздаёт запись; проверь
  ветку `if (!connection)` в роуте (upsert должен вызываться только при отсутствии записи).
- Кнопка «Копировать» ничего не делает → в Telegram WebView/HTTP `navigator.clipboard` может быть
  недоступен; смотри вкладку Console и `src/components/miniapp/CopyField.tsx` (фолбэк не должен падать).
- В консоли сервера мелькнул текст токена → ищи `console.*` с телом запроса; в норме их нет.
- В `ig_connections` `*_enc` равны введённому тексту → шифрование не применилось, смотри
  `src/lib/db/igConnections.ts` (`encrypt()` должен вызываться внутри репозитория).

## Сопоставление с критериями приёмки
- **Секреты никогда не возвращаются с бэкенда и не пишутся в логи** → проверки 3, 8; негатив 1.
- **webhookUrl/verifyToken корректны и копируются в один тап** → проверки 1, 2, 4.
- **Повторный вход показывает статус и маскированные поля** → проверки 6, 7.
