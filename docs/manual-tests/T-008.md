# Ручной тест T-008 · LLM-клиент

## Что сделано
- Добавлен единый LLM-клиент `src/lib/llm/client.ts` для OpenAI-compatible API/OpenRouter.
- Клиент поддерживает ретраи 429/5xx, таймаут 30 секунд, JSON-режим с повтором при невалидном JSON и учёт usage в `usage_stats`.
- Добавлен демо-скрипт для проверки JSON-режима вручную.

## Подготовка
- ветка: `git fetch && git checkout ticket/T-008 && pnpm install`
- env: заполнить `.env.local` значениями из `.env.example`:
  - `LLM_BASE_URL` — например `https://openrouter.ai/api/v1`;
  - `LLM_API_KEY` — ключ OpenRouter;
  - `LLM_MODEL_CLASSIFY` и `LLM_MODEL_DRAFT` — доступные модели, например `openai/gpt-4o-mini`;
  - `SUPABASE_URL` и `SUPABASE_SECRET_KEY` — dev-проект Supabase, если проверяется запись usage;
  - остальные обязательные переменные можно заполнить dev/test значениями; `ENCRYPTION_KEY` можно сгенерировать через `pnpm run gen:key`.
- сервисы: для проверки usage применить миграции на dev-проект Supabase через `pnpm db:push` и иметь существующий `tenant_id` из таблицы `tenants`.

## Проверки (каждая: Действие → Ожидаемо)
1. Выполнить `pnpm lint && pnpm typecheck` → обе команды завершаются без ошибок.
2. Выполнить `rg "from 'openai'|from \"openai\"|require\('openai'\)|require\(\"openai\"\)" -n src app scripts --glob "!src/lib/llm/**"` → команда не выводит совпадений.
3. Без `tenant_id` выполнить `pnpm tsx scripts/manual/T-008-demo.ts` → скрипт печатает `T-008 OK: JSON parsed successfully`, объект `data` проходит Zod-схему, в `usage` есть числовые `inTok` и `outTok`.
4. С учётом usage выполнить `pnpm tsx scripts/manual/T-008-demo.ts <tenant_id>` → скрипт завершается успешно, а в Supabase в таблице `usage_stats` для сегодняшнего дня увеличиваются `llm_calls`, `tokens_in`, `tokens_out`.
5. Временно указать в `LLM_MODEL_CLASSIFY` модель/провайдера, который возвращает 429 или 5xx, либо исчерпанную квоту → в логах видно, что клиент делает повторные попытки с задержкой; при последующем успехе скрипт завершается успешно.

## Негативные проверки
1. Указать неверный `LLM_API_KEY` и выполнить `pnpm tsx scripts/manual/T-008-demo.ts` → скрипт завершается с ошибкой `LlmError: request_failed:401` или другим 4xx без серии повторов.
2. Указать несуществующую модель в `LLM_MODEL_CLASSIFY` → скрипт завершается `LlmError` для 4xx-ошибки провайдера без ретраев.
3. Для проверки JSON-валидации временно заменить Zod-схему в `scripts/manual/T-008-demo.ts` на несовместимую, например требовать поле `missing` → после одного повторного запроса скрипт завершается `LlmError: bad_json`.

## Как понять, что тикет НЕ работает
- `pnpm typecheck` ругается на `openai` или `src/lib/llm/client.ts` → SDK/типы подключены некорректно.
- `rg ...openai... -n src app scripts --glob "!src/lib/llm/**"` показывает импорты → нарушена единственная точка вызова LLM.
- При 4xx-ошибке видно несколько повторных запросов → нарушена логика ретраев.
- При успешном вызове с `tenant_id` не растёт `usage_stats` → проверить логи приложения на сообщение `LLM usage accounting failed`, корректность `SUPABASE_*` и наличие tenant.
