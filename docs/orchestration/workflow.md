# Регламент разработки (общий для Claude Code и Codex)

Проект собирается по тикетам `docs/tickets/T-0XX.md`. Работа идёт в облаке, изменения — в GitHub.
Автоматических тестов НЕТ: каждый тикет принимает ЧЕЛОВЕК, вручную, по инструкции, которую агент
обязан написать. Читай `docs/tickets/OVERRIDES.md` — он корректирует тексты тикетов.

## Статусы тикета (state/progress.json)
pending → in_progress → awaiting_merge → done | failed | blocked | skipped
- awaiting_merge: код в ветке, PR открыт, инструкция ручного теста написана. Ждём человека.
- done: ставится ТОЛЬКО после фактического попадания ветки тикета в main (человек протестировал и влил).
- Поля: status, attempts, branch, pr (номер/URL или null), deps, notes.
- progress.json правится атомарно, по одному тикету, и коммитится в ветку тикета вместе с кодом.

## Процедура RESUME (начало каждой сессии, любой агент)
1. `git fetch origin && git checkout main && git pull`.
2. Для каждого тикета в awaiting_merge: ветка влита в main (`git branch -r --merged origin/main`
   или история)? → статус done. Ветка удалена без мержа / PR закрыт → спроси человека.
3. Есть in_progress? Значит прошлая сессия оборвалась: осмотри ветку тикета; работа фрагментарна →
   доложи человеку и предложи (а) продолжить с места обрыва, (б) начать тикет заново с чистой ветки.
   Сам не решай.
4. Обновлённый progress.json закоммить в main отдельным коммитом `chore: sync progress`.
5. Доложи: done/awaiting_merge/следующий доступный (по deps, меньший номер).

## Конвейер одного тикета
1. deps-check: все зависимости done. Не все → blocked/стоп.
2. Ветка: `git checkout -b ticket/T-0XX` от свежего main. Статус in_progress (коммит в ветку).
3. РЕАЛИЗАЦИЯ (Claude: сабагент implementer; Codex: роль implementer):
   строго по файлу тикета + plan.md §5 + OVERRIDES. Только файлы, вытекающие из тикета.
4. САМОПРОВЕРКА: `pnpm lint && pnpm typecheck` зелёные (это НЕ тесты, это гигиена — обязательна).
5. РЕВЬЮ (Claude: сабагент reviewer; Codex: роль reviewer). Чек-лист:
   - каждый критерий приёмки тикета (в редакции OVERRIDES) закрыт кодом;
   - соглашения §5: env только через env.ts; БД только через src/lib/db; секреты через crypto;
     промпты в prompts.ts; пользовательские строки в HTML — только через escapeHTML;
   - нет изменений вне скоупа, нет мёртвого кода, нет секретов в коде/логах.
   Вердикт approved | changes_required(список: файл:строка → что → как чинить).
   changes_required → исправление → повторное ревью. Максимум 2 цикла, дальше failed.
6. ИНСТРУКЦИЯ РУЧНОГО ТЕСТА: создай `docs/manual-tests/T-0XX.md` по шаблону
   `docs/orchestration/manual-testing.md`. Это обязательный артефакт тикета — без него PR не открывать.
7. ФИНАЛИЗАЦИЯ: progress.json → awaiting_merge (+branch), строка в docs/build-log.md,
   коммит(ы) с префиксом `T-0XX:`, push ветки, PR в main. Тело PR: краткое описание +
   ПОЛНЫЙ текст инструкции ручного теста + чек-лист критериев приёмки.
8. СТОП. Следующий тикет не начинать, пока человек не вольёт PR (гейт ручного тестирования).
   Исключение: человек явно разрешил параллелить независимые по deps тикеты — тогда каждый
   в своей ветке от main.

## Failed и эскалация
2 цикла ревью или неустранимая проблема → status failed + notes с полным диагнозом → стоп,
подробное сообщение человеку: что не вышло, гипотезы, варианты (уточнить тикет / вручную / пропустить).

## Передача смены между агентами (Claude ↔ Codex)
Всё состояние — в репозитории. Перед завершением сессии: доведи текущий ШАГ конвейера до конца
(не бросай посреди реализации без коммита), запушь ветку, обнови progress.json (notes: на каком
шаге остановился), напиши человеку сводку из 3-5 строк. Принимающий агент стартует с RESUME.

## Запреты (все агенты)
- Не менять docs/plan.md, docs/tickets/** (кроме OVERRIDES — и его только человек), docs/orchestration/**.
- Не коммитить в main напрямую (кроме chore: sync progress на шаге RESUME).
- Не читать и не логировать .env*, не вписывать секреты в код, PR, manual-tests.
- Не ставить статус done самовольно — done означает «человек принял».
- Не добавлять зависимости, не предусмотренные тикетом, без вопроса человеку.
- Не выполнять деструктивные команды (rm -rf, force-push, db reset).
